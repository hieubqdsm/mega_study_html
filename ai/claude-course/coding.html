01 Claude Code là gì — và khác gì ô chat

Bạn vẫn có thể code bằng cách dán đoạn code vào Claude.ai. Nhưng để khai thác hết khả năng, hãy hiểu Claude Code.

Claude Code là Claude chạy trong terminal của bạn, ngay bên trong thư mục dự án. Khác biệt cốt lõi so với việc copy-paste vào ô chat:

ViệcDán code vào Claude.aiClaude Code
Đọc file dự ánBạn phải tự dán từng fileTự đọc, tự tìm file liên quan
Sửa codeBạn copy kết quả về dán lạiSửa thẳng vào file
Chạy lệnh / testKhôngChạy được, đọc kết quả, tự sửa
Hiểu cả dự ánGiới hạn ở thứ bạn dánKhám phá toàn bộ codebase

Nói ngắn: ô chat hợp cho một đoạn code lẻ hoặc câu hỏi khái niệm. Claude Code hợp khi bạn làm việc trong một dự án thật — và đó là phần lớn công việc lập trình.

💡

Claude Code có ở terminal, ứng dụng máy tính, bản web và tiện ích cho VS Code / JetBrains. Cùng một năng lực, chọn nơi bạn thấy thoải mái. Phần còn lại của module dùng được cho mọi giao diện.

02 Khởi động dự án đúng cách

5 phút thiết lập đầu tiên quyết định chất lượng mọi tương tác sau đó.

Lập file CLAUDE.md — trí nhớ của dự án

Đặt một file CLAUDE.md ở thư mục gốc. Claude Code đọc nó tự động ở mỗi phiên làm việc. Đây là nơi ghi mọi điều bạn không muốn lặp lại:

Mẫu CLAUDE.md
# Dự án: [tên]

## Tổng quan
[Ứng dụng làm gì, cho ai, trong 2–3 câu]

## Công nghệ
- Ngôn ngữ / framework: [...]
- Cơ sở dữ liệu: [...]
- Quản lý gói: [npm / pip / ...]

## Lệnh quan trọng
- Cài đặt: `[lệnh]`
- Chạy dev:  `[lệnh]`
- Chạy test: `[lệnh]`
- Build:     `[lệnh]`

## Quy ước
- Phong cách code: [vd: 2 dấu cách, dùng dấu nháy đơn]
- Đặt tên: [quy ước của bạn]
- Cấu trúc thư mục: [giải thích ngắn]

## Điều CẦN TRÁNH
- Không sửa thư mục /legacy.
- Không thêm thư viện mới khi chưa hỏi.
- Luôn viết test cho logic nghiệp vụ.

Nếu dự án đã có sẵn, bạn có thể nhờ Claude tự soạn nháp file này:

Prompt
Khảo sát dự án này và soạn nháp file CLAUDE.md gồm: tổng quan,
công nghệ sử dụng, các lệnh build/test/run phát hiện được, và quy ước
code mà bạn suy ra từ chính source. Đánh dấu [?] ở chỗ bạn không chắc
để tôi điền nốt.

Mở đầu phiên làm việc bằng mục tiêu, không phải mệnh lệnh

Thay vì lao thẳng vào "sửa hàm này", hãy mô tả kết quả mong muốn và để Claude khảo sát trước. Module Nền tảng gọi đây là chia nhỏ nhiệm vụ — áp dụng đầu tiên ở bước hiểu vấn đề.

03 Viết tính năng mới theo đặc tả

Sai lầm phổ biến: yêu cầu mơ hồ → nhận code chạy được nhưng sai ý. Hãy đặc tả trước khi code.

Quy trình 4 bước cho một tính năng

  1. Mô tả bằng hành vi, không bằng cách làmNói tính năng phải làm gì và trông ra sao, để Claude tự đề xuất cách hiện thực rồi bạn duyệt.
  2. Yêu cầu kế hoạch trước khi viết code"Trước khi code, hãy mô tả kế hoạch: file nào cần sửa/tạo, các bước, rủi ro." Bạn duyệt kế hoạch — rẻ hơn duyệt 300 dòng code.
  3. Cho hiện thực theo từng phầnLàm từng phần một, kiểm tra giữa chừng. Tránh "đổ" cả tính năng trong một lượt.
  4. Yêu cầu chạy thử & tự kiểm"Chạy test và sửa cho tới khi xanh. Báo lại đã đổi những gì."
Prompt — đặc tả tính năng
Tôi muốn thêm tính năng: [tên tính năng].

HÀNH VI MONG MUỐN
- Khi người dùng [hành động], hệ thống [kết quả].
- Trường hợp biên: [vd: dữ liệu rỗng, lỗi mạng, quyền hạn].
- Không được phá: [chức năng hiện có cần giữ nguyên].

RÀNG BUỘC
- Theo đúng quy ước trong CLAUDE.md.
- Không thêm thư viện mới khi chưa hỏi tôi.

TRƯỚC KHI VIẾT CODE:
1. Tìm và liệt kê các file liên quan.
2. Trình bày kế hoạch hiện thực theo bước.
3. Nêu điểm chưa rõ cần tôi quyết.
Chờ tôi duyệt kế hoạch rồi mới code.
⚠️

Câu "chờ tôi duyệt kế hoạch rồi mới code" là cái van an toàn quan trọng nhất. Nó giúp bạn bắt sai lệch khi sửa còn rẻ — thay vì sau khi đã có cả đống code sai hướng.

04 Đọc & hiểu một codebase lạ

Đây là nơi Claude tiết kiệm thời gian nhiều nhất — và bị dùng ít nhất. Tiếp nhận dự án mới, kế thừa code cũ, hay đọc thư viện mã nguồn mở.

Prompt — bản đồ tổng quan
Tôi mới tiếp nhận dự án này. Hãy giúp tôi định hướng:

1. Ứng dụng này làm gì? (suy ra từ source)
2. Sơ đồ kiến trúc cấp cao: các thành phần chính và cách chúng nối nhau.
3. Một yêu cầu của người dùng đi qua hệ thống theo đường nào?
4. 5 file quan trọng nhất nên đọc đầu tiên, và vì sao.
5. Có chỗ nào trông như "nợ kỹ thuật" hoặc dễ gây nhầm không?

Trình bày cho người mới vào dự án, dùng đường dẫn file thật.

Sau bức tranh tổng thể, đào sâu một phần cụ thể:

Prompt — giải thích một luồng
Giải thích chi tiết luồng [vd: đăng nhập / thanh toán] hoạt động ra sao.
- Bắt đầu từ điểm nào, kết thúc ở đâu.
- Mỗi bước, chỉ rõ file và hàm xử lý.
- Vẽ sơ đồ tuần tự bằng văn bản.
- Chỉ ra chỗ dễ hỏng hoặc cần thận trọng khi sửa.
💡

Mẹo: bảo Claude "giải thích như cho một lập trình viên junior mới vào nhóm". Bạn nhận được lời giải thích không bỏ sót bước ngầm — đúng thứ bạn cần khi codebase còn lạ.

05 Debug có hệ thống

Đừng dán lỗi kèm câu "sửa giúp". Hãy biến Claude thành một người gỡ lỗi có phương pháp.

Cung cấp đủ "hiện trường"

Một báo cáo lỗi tốt cho Claude gồm: hành vi mong đợi, hành vi thực tế, thông báo lỗi đầy đủ, các bước tái hiện, và những gì bạn đã thử. Thiếu phần nào, Claude phải đoán phần đó.

Prompt — debug
Có một lỗi cần xử lý.

MONG ĐỢI: [điều đáng lẽ phải xảy ra]
THỰC TẾ:  [điều đang xảy ra]
LỖI / LOG:
"""
[dán nguyên văn, đầy đủ]
"""
TÁI HIỆN: [các bước để lỗi xuất hiện]
ĐÃ THỬ:  [những gì đã làm và kết quả]

Cách làm tôi muốn:
1. Nêu 3–4 giả thuyết về nguyên nhân, xếp theo khả năng.
2. Với mỗi giả thuyết, chỉ cách kiểm chứng nhanh.
3. Tìm nguyên nhân gốc — ĐỪNG vá triệu chứng.
4. Đề xuất bản sửa nhỏ gọn nhất, giải thích vì sao nó đúng.

Khi lỗi "ngoan cố"

Nếu sửa vài lần vẫn chưa xong, đừng vá tiếp lên vết vá. Hãy lùi lại:

Prompt — lùi lại một bước
Ta đã thử vài cách mà lỗi vẫn còn. Dừng việc sửa lại.
Hãy chèn log ở các điểm nghi ngờ để xác nhận giả định nào sai.
Liệt kê các log cần thêm và dữ liệu cần quan sát ở mỗi điểm.
Tôi sẽ chạy và đưa kết quả lại cho bạn.
⚠️

Cảnh giác với "bản sửa làm test xanh mà không hiểu vì sao". Nếu Claude sửa được nhưng không giải thích nổi nguyên nhân gốc, hãy bắt nó giải thích lại. Lỗi không hiểu sẽ quay về.

06 Refactor quy mô lớn

Đổi tên xuyên dự án, tách module, nâng cấp thư viện, dọn nợ kỹ thuật — việc tẻ nhạt và dễ sai, đúng kiểu việc nên giao cho Claude.

Nguyên tắc: refactor không đổi hành vi

Refactor là cải thiện cấu trúc mà giữ nguyên hành vi. Vì vậy, luôn đảm bảo có lưới an toàn trước khi bắt đầu:

  1. Chốt lưới an toànPhải có test, hoặc một cách kiểm tra hành vi trước/sau. Chưa có test? Bảo Claude viết test cho hành vi hiện tại trước.
  2. Yêu cầu kế hoạch & phạm vi"Liệt kê mọi file bị ảnh hưởng, chia thành các bước nhỏ độc lập." Duyệt phạm vi trước.
  3. Làm từng bước, kiểm sau mỗi bướcMỗi bước xong là chạy test. Lỗi xuất hiện ở đâu thì biết do bước nào.
  4. So sánh hành vi cuối cùng"Xác nhận hành vi sau refactor giống hệt trước. Nếu có khác biệt, nêu rõ."
Prompt — refactor an toàn
Tôi muốn refactor: [mô tả, vd: tách UserService thành các phần nhỏ].

MỤC TIÊU: [vì sao — dễ test hơn / dễ đọc hơn / ...]
BẤT BIẾN: hành vi quan sát được từ bên ngoài KHÔNG đổi.

Cách làm:
1. Trước tiên, kiểm tra mức độ phủ test của vùng này. Nếu mỏng, đề xuất
   test cần bổ sung trước khi đụng vào.
2. Lập kế hoạch refactor theo các bước nhỏ, mỗi bước tự đứng vững và
   test vẫn xanh sau mỗi bước.
3. Chờ tôi duyệt. Sau đó làm từng bước, dừng lại sau mỗi bước báo cáo.

07 Viết test & review code

Hai việc lập trình viên hay né — và Claude làm rất tốt nếu được hướng đúng.

Viết test có ý nghĩa, không phải test cho có

Bảo "viết test" thì Claude dễ ra những test tầm thường. Hãy yêu cầu nó nhắm vào nơi lỗi thật sự ẩn náu:

Prompt — viết test
Viết test cho [hàm/module].

Ưu tiên các nhóm sau, không chỉ trường hợp thuận lợi:
- Trường hợp biên: rỗng, null, 0, số âm, chuỗi rất dài.
- Trường hợp lỗi: đầu vào sai, phụ thuộc bên ngoài thất bại.
- Quy tắc nghiệp vụ: mọi nhánh điều kiện đều có test riêng.

Mỗi test đặt tên mô tả rõ tình huống. Theo đúng phong cách test
sẵn có trong dự án. Cuối cùng, chỉ ra trường hợp nào BẠN cho rằng
vẫn chưa được phủ.

Dùng Claude làm người review code

Trước khi mở pull request, cho Claude review trước. Nó không thay người review, nhưng bắt được phần lớn lỗi vặt và điểm cần bàn:

Prompt — review code
Review phần thay đổi tôi sắp commit. Đóng vai một senior khó tính
nhưng xây dựng. Phân loại nhận xét theo mức độ:

[CHẶN]    Lỗi, lỗ hổng bảo mật, dữ liệu hỏng — phải sửa.
[NÊN]     Vấn đề thiết kế, trường hợp biên bị bỏ sót.
[GÓP Ý]   Dễ đọc hơn, đặt tên, phong cách.

Với mỗi điểm: nêu vị trí (file:dòng), vì sao quan trọng, và cách sửa
gợi ý. Nếu code đã ổn ở mảng nào, nói rõ — đừng bịa vấn đề.
💡

Việc phân loại CHẶN / NÊN / GÓP Ý giúp bạn không bị ngợp: sửa ngay nhóm CHẶN, cân nhắc nhóm NÊN, để dành nhóm GÓP Ý. Cùng cách phân loại này dùng được khi review bài của người khác.

08 Kỹ thuật nâng cao

Khi đã quen, bốn năng lực sau nâng Claude Code lên một tầm khác.

Chế độ lập kế hoạch (Plan mode)

Một chế độ mà Claude chỉ khảo sát và lập kế hoạch, không sửa file. Dùng cho việc lớn hoặc rủi ro cao: bạn duyệt kế hoạch hoàn chỉnh rồi mới cho phép thực thi. Đây là "yêu cầu kế hoạch trước" được chuẩn hóa thành một chế độ.

Subagent — chia việc cho các trợ lý phụ

Với việc lớn nhiều nhánh, Claude Code có thể giao việc con cho các "subagent" chạy song song — ví dụ một agent dò toàn bộ codebase trong khi agent chính tiếp tục. Bạn không phải cấu hình gì phức tạp; chỉ cần biết với nhiệm vụ lớn, có thể yêu cầu Claude "khảo sát rộng" và nó tự phân bổ.

MCP — nối Claude với công cụ bên ngoài

MCP (Model Context Protocol) cho phép Claude Code kết nối tới hệ thống ngoài: cơ sở dữ liệu, hệ thống quản lý lỗi, tài liệu nội bộ, dịch vụ đám mây. Khi đã nối, bạn có thể nhờ Claude "đọc vé lỗi này và sửa", hay "truy vấn database staging để kiểm tra giả thuyết". Bắt đầu từ một kết nối bạn dùng nhiều nhất.

Tự động hóa & việc lặp lại

Việc kỹ thuật lặp đi lặp lại — phân loại lỗi mới, cập nhật phụ thuộc, sinh báo cáo — có thể được Claude Code thực hiện theo lịch hoặc theo sự kiện. Khi một quy trình thủ công đã ổn định, hãy nghĩ tới việc biến nó thành tác vụ tự động.

🎯

Đừng vội dùng hết tính năng nâng cao. Thành thạo vòng lặp cốt lõi trước: mô tả → kế hoạch → duyệt → thực thi từng phần → kiểm. Tính năng nâng cao chỉ làm vòng lặp đó nhanh hơn, không thay thế nó.

★ Bài tập module Coding
  1. Lập CLAUDE.mdCho một dự án thật của bạn, tạo file CLAUDE.md. Mở phiên mới và kiểm Claude đã nắm đúng bối cảnh chưa.
  2. Lập bản đồ codebaseChọn một dự án bạn chưa rành (của bạn hoặc mã nguồn mở). Dùng prompt "bản đồ tổng quan" và đối chiếu với hiểu biết của bạn.
  3. Một tính năng theo quy trình 4 bướcChọn một tính năng nhỏ. Bắt buộc đi qua: đặc tả → kế hoạch → duyệt → thực thi từng phần. Không bỏ bước duyệt.
  4. Một phiên debug có phương phápLần lỗi tới, dùng prompt debug đầy đủ "hiện trường". So với cách bạn thường làm.
  5. Review trước khi commitLần thay đổi tới, cho Claude review bằng thang CHẶN / NÊN / GÓP Ý trước khi mở PR.
⚠️

Lỗi thường gặp cần tránh: chấp nhận code mà không đọc · bỏ bước duyệt kế hoạch để "cho nhanh" · để Claude thêm thư viện tùy ý · refactor khi chưa có lưới test · tin "test đã xanh" mà không hiểu bản sửa.
