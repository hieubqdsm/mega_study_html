CH 1 PostgreSQL là gì & vì sao chọn

Bản chất RDBMS, lịch sử, triết lý đúng đắn, và khi nào nên/không nên dùng.

Cơ sở dữ liệu quan hệ (RDBMS) là gì?

Dữ liệu được tổ chức thành bảng (table) gồm các dòng (row/record) và cột (column). Các bảng liên kết với nhau qua khóa (key). Bạn thao tác bằng ngôn ngữ SQL (Structured Query Language). Đây là mô hình thống trị suốt 40+ năm vì nó đảm bảo dữ liệu nhất quán, không trùng lặp, và truy vấn linh hoạt.

PostgreSQL (đọc là "post-gress-Q-L", thường gọi tắt "Postgres") ra đời năm 1986 tại UC Berkeley, là một object-relational DBMS: ngoài quan hệ thuần còn hỗ trợ kiểu dữ liệu tự định nghĩa, kế thừa bảng, hàm trong nhiều ngôn ngữ. Nó nổi tiếng vì tuân thủ chuẩn SQL nghiêm ngặt và ưu tiên tính đúng đắn của dữ liệu hơn là "chạy được cho xong".

ACID — 4 đảm bảo của một giao dịch Atomicity (nguyên tử: all-or-nothing) · Consistency (nhất quán: luôn thỏa ràng buộc) · Isolation (cô lập: giao dịch song song không giẫm chân nhau) · Durability (bền vững: đã commit là không mất kể cả mất điện). PostgreSQL đảm bảo cả 4 ở mặc định.

CH 2 Cài đặt & công cụ làm việc

Cài Postgres trên máy, dùng psql, và các GUI phổ biến.

Cài đặt

  • Windows / macOS: tải bộ cài từ postgresql.org/download (kèm pgAdmin). macOS còn có Postgres.app rất tiện.
  • Linux (Debian/Ubuntu): sudo apt install postgresql
  • Docker (khuyên dùng cho dev): docker run -e POSTGRES_PASSWORD=secret -p 5432:5432 postgres:16
Khái niệm cụm (cluster)Một server PostgreSQL chứa nhiều database; mỗi database chứa nhiều schema (mặc định là public); mỗi schema chứa bảng, view, function... Người dùng đăng nhập bằng role. Cổng mặc định: 5432.

psql — dòng lệnh chính thức

psql là client terminal. Kết nối: psql -h localhost -U postgres -d mydb. Các lệnh "meta" bắt đầu bằng dấu gạch chéo ngược:

psql-meta.txt
\l            -- liệt kê databases
\c mydb       -- chuyển sang database mydb
\dt          -- liệt kê bảng (tables)
\d employees -- mô tả cấu trúc bảng employees
\du          -- liệt kê roles/users
\dn          -- liệt kê schemas
\timing      -- bật đo thời gian truy vấn
\x           -- chế độ hiển thị dọc (expanded), dễ đọc
\i file.sql  -- chạy file SQL
\q           -- thoát

Các lệnh \ là của riêng psql, không phải SQL — nên không chạy được trong playground trình duyệt ở đây.

Quy tắc thực dụng
  • Tiền/giá → NUMERIC (KHÔNG dùng float vì sai số làm tròn).

⚖️ Kiểu dữ liệu: Postgres vs MySQL vs SQLite Postgres: kho kiểu phong phú nhất (array, jsonb, range, network, custom type, enum thật). MySQL: có ENUM, JSON nhưng không có array hay range bản địa; phân biệt nhiều kiểu integer signed/unsigned. SQLite: nổi tiếng "type affinity" — cột gần như không ép kiểu cứng, bạn có thể nhét chuỗi vào cột INTEGER. Linh hoạt nhưng dễ lỗi âm thầm; Postgres thì kiểm soát kiểu rất chặt.

CH 23 MVCC & khóa (Concurrency)

Bí mật giúp Postgres cho nhiều người đọc/ghi cùng lúc mà không "kẹt".

MVCC (Multi-Version Concurrency Control) là cơ chế cốt lõi: mỗi lần UPDATE/DELETE, Postgres không ghi đè dòng cũ mà tạo phiên bản mới, giữ phiên bản cũ cho các transaction đang đọc. Hệ quả vàng: người đọc không bao giờ chặn người ghi, và ngược lại. Mỗi transaction thấy một "ảnh chụp" (snapshot) nhất quán của dữ liệu.

Hệ quả của MVCC
  • Đọc không cần khóa → đọc đồng thời cực tốt.
  • Sinh ra "dead tuples" (phiên bản cũ chết) → cần VACUUM dọn (chương 24).
  • Số transaction (XID) tăng dần → cần tránh "transaction ID wraparound".

Khóa khi ghi cùng dòng

Hai transaction cùng sửa một dòng thì cái sau phải chờ cái trước commit/rollback. Để chủ động khóa khi đọc nhằm cập nhật an toàn (chống race condition), dùng SELECT ... FOR UPDATE:

for_update.sql
BEGIN;
  -- khóa dòng để không ai sửa xen vào giữa
  SELECT balance FROM accounts WHERE username = 'alice' FOR UPDATE;
  UPDATE accounts SET balance = balance - 100 WHERE username = 'alice';
COMMIT;

⚖️ MVCC: Postgres vs các DB khácPostgres lưu phiên bản cũ ngay trong bảng (heap) → cần VACUUM. Oracle/MySQL-InnoDB dùng "undo log/rollback segment" riêng → ít bloat hơn nhưng cơ chế phức tạp khác. SQLite đơn giản hóa bằng khóa toàn database khi ghi (chỉ 1 writer tại một thời điểm), đánh đổi concurrency lấy sự đơn giản.

CH 24 VACUUM & bảo trì

Dọn rác MVCC, cập nhật thống kê, giữ database khỏe.

Vì MVCC để lại "dead tuples", bảng sẽ phình (bloat) nếu không dọn. VACUUM thu hồi không gian các dòng chết để tái sử dụng; ANALYZE cập nhật thống kê cho query planner; VACUUM FULL nén bảng thật sự (khóa bảng — dùng cẩn thận).

vacuum.sql
VACUUM employees;            -- thu hồi dead tuples
ANALYZE employees;           -- cập nhật thống kê cho planner
VACUUM (VERBOSE, ANALYZE) employees;
VACUUM FULL employees;       -- nén triệt để (KHÓA bảng!)
REINDEX TABLE employees;     -- xây lại index bị phình
AutovacuumPostgres có tiến trình autovacuum tự động chạy nền — hầu hết trường hợp bạn không cần VACUUM thủ công. Nhưng với bảng ghi/cập nhật cực nhiều, cần tinh chỉnh ngưỡng autovacuum để tránh bloat tích tụ.
Dấu hiệu cần chú ýTruy vấn chậm dần, kích thước bảng phình bất thường, planner chọn sai kế hoạch (thống kê cũ) → thường liên quan VACUUM/ANALYZE chưa kịp. Theo dõi qua pg_stat_user_tables (cột n_dead_tup, last_autovacuum).

CH 29 Scaling & chọn đúng database

Tổng kết: khi nào Postgres tỏa sáng, khi nào nên chọn thứ khác.

Thang scale một Postgres

  1. Tối ưu truy vấn & index — rẻ nhất, hiệu quả nhất.
  2. Tăng phần cứng (vertical scaling) — RAM, SSD NVMe, CPU.
  3. Connection pooling (PgBouncer) — Postgres tốn tài nguyên mỗi kết nối.
  4. Caching (Redis) cho dữ liệu nóng; materialized view cho báo cáo.
  5. Read replica — phân tải đọc.
  6. Partitioning cho bảng khổng lồ.
  7. Sharding (Citus) — bước cuối, phức tạp nhất.

Xem gợi ý lời giải & trade-off

1. Khóa chính: BIGINT identity cho bảng nội bộ (nhỏ gọn, index hiệu quả, tuần tự tốt cho B-tree). Dùng UUID khi id bị lộ ra ngoài/URL hoặc sinh phân tán nhiều nơi (đổi lại index cồng kềnh hơn — cân nhắc UUIDv7 để gần tuần tự).

2. Multi-tenant: mọi bảng có cột tenant_id; bật Row Level Security với policy lọc theo tenant_id = phiên hiện tại. Tầng ứng dụng set SET app.tenant_id. RLS là lưới an toàn cuối cùng kể cả khi code quên thêm WHERE.

3. Partition + index: partition activity_logs theo RANGE thời gian (tháng/năm) → truy vấn theo khoảng thời gian chỉ chạm partition liên quan, và xóa dữ liệu cũ = DROP partition (cực nhanh, không bloat). Composite index (tenant_id, created_at DESC) khớp đúng mẫu truy vấn phổ biến nhất.

4. Báo cáo: đẩy truy vấn đọc nặng sang read replica; dùng materialized view (refresh định kỳ) cho dashboard; cân nhắc bảng tổng hợp (rollup) cập nhật bằng job. Tránh chạy báo cáo ad-hoc trên primary giờ cao điểm.

5. Scaling: theo thứ tự — tối ưu query/index → tăng phần cứng → PgBouncer (pooling) → Redis cache → read replica → partition → cuối cùng sharding (Citus). Backup: pgBackRest + PITR (base backup + WAL). HA: streaming replication + tự động failover (Patroni). Theo dõi bằng pg_stat_statements.

EXPLAIN chỉ ước lượng kế hoạch; EXPLAIN ANALYZE chạy thật và báo thời gian thực tế.
L3
MVCC là gì?
ĐÁP ÁN
Multi-Version Concurrency Control: tạo phiên bản mới thay vì ghi đè → người đọc không chặn người ghi. Sinh dead tuples cần VACUUM.
L3
WAL dùng để làm gì?
ĐÁP ÁN
Write-Ahead Log: ghi nhật ký thay đổi trước khi áp vào dữ liệu → đảm bảo Durability, nền tảng cho recovery & replication.
L3
Khi nào KHÔNG nên dùng Postgres?
ĐÁP ÁN
App nhúng offline → SQLite; cache → Redis; OLAP/log cực lớn → ClickHouse; cần scale ngang ghi cực lớn ngay → kiến trúc phân tán/MongoDB.

📋 Cheatsheet

Tra cứu nhanh cú pháp & lệnh hay dùng.

DDL — định nghĩa cấu trúc

ddl.sql
CREATE TABLE t (id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY, name TEXT NOT NULL);
ALTER TABLE t ADD COLUMN age INT;
ALTER TABLE t RENAME COLUMN age TO tuoi;
DROP TABLE IF EXISTS t CASCADE;
CREATE INDEX idx ON t(name);
TRUNCATE t;                  -- xóa sạch nhanh (không WHERE)

DML — thao tác dữ liệu

dml.sql
INSERT INTO t (name) VALUES ('a'),('b') RETURNING id;
UPDATE t SET name='x' WHERE id=1;
DELETE FROM t WHERE id=1;
INSERT INTO t (id,name) VALUES (1,'a')
  ON CONFLICT (id) DO UPDATE SET name=EXCLUDED.name;

SELECT — truy vấn

select.sql
SELECT col, COUNT(*) FROM t
WHERE x BETWEEN 1 AND 10 AND y IN (1,2) AND z ILIKE '%ab%'
GROUP BY col HAVING COUNT(*) > 1
ORDER BY col DESC NULLS LAST
LIMIT 10 OFFSET 20;

JOIN & CTE & Window

advanced.sql
SELECT a.*, b.* FROM a JOIN b ON a.id=b.a_id;     -- INNER
SELECT a.* FROM a LEFT JOIN b ON ... WHERE b.id IS NULL; -- anti-join

WITH x AS (SELECT ...) SELECT * FROM x;        -- CTE

ROW_NUMBER() OVER (PARTITION BY g ORDER BY v DESC)    -- window

JSONB

jsonb.sql
data->'key'      -- lấy JSONB con
data->>'key'     -- lấy TEXT
data#>>'{a,b}'  -- đường dẫn lồng → TEXT
data @> '{"k":1}' -- chứa
data ? 'key'      -- tồn tại key
CREATE INDEX ... USING GIN (data);

Transaction & bảo trì

tx.sql
BEGIN; ... COMMIT;  |  ROLLBACK;  |  SAVEPOINT s; ROLLBACK TO s;
SELECT ... FOR UPDATE;          -- khóa dòng

EXPLAIN ANALYZE SELECT ...;     -- xem kế hoạch thực thi
