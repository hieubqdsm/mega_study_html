Giới Thiệu Git & Version Control

1.1. Version Control System (VCS) là gì?

Version Control System (Hệ thống quản lý phiên bản) là phần mềm giúp theo dõi và quản lý thay đổi trong source code theo thời gian. Nó cho phép nhiều người làm việc cùng lúc trên 1 dự án mà không "đè" code lên nhau.

💡 Hình dung đơn giản
VCS giống như Google Docs cho code: bạn thấy ai sửa gì, khi nào, có thể quay lại bất kỳ phiên bản nào trong quá khứ, và làm việc song song với nhiều người mà không xung đột.

1.2. Lịch Sử Ra Đời Git

Git được tạo ra bởi Linus Torvalds (cha đẻ Linux) vào năm 2005 sau khi BitKeeper - VCS đang được dùng cho Linux kernel - rút lại quyền sử dụng miễn phí.

  • 10 ngày - thời gian Linus viết phiên bản đầu của Git
  • "Git" = đồ ngu/đần (tiếng Anh-Anh slang) - Linus nói đùa "I name all my projects after myself"
  • Mục tiêu thiết kế: tốc độ, thiết kế đơn giản, hỗ trợ phát triển phi tuyến tính, fully distributed

1.3. Centralized vs Distributed VCS

📡 Centralized (SVN, CVS)

  • Có 1 server trung tâm chứa toàn bộ lịch sử
  • Client chỉ lấy phiên bản hiện tại
  • Cần kết nối mạng để commit/log
  • Server down → cả team đứng
  • Đơn giản, dễ hiểu cho người mới
VS

🌐 Distributed (Git, Mercurial)

  • Mỗi máy đều có full copy của repository
  • Commit/branch offline thoải mái
  • Backup tự nhiên (mỗi clone là 1 backup)
  • Workflow linh hoạt (push lên đâu cũng được)
  • Học khó hơn nhưng mạnh hơn nhiều

2.2. Cấu Hình Lần Đầu

Trước khi commit code, Git cần biết bạn là ai để ghi vào lịch sử.

$ git config --global user.name "Nguyen Van A" $ git config --global user.email "vana@example.com" # Đặt editor mặc định (mở khi cần viết commit message dài) $ git config --global core.editor "code --wait" # Đổi tên branch mặc định từ "master" thành "main" $ git config --global init.defaultBranch main # Kiểm tra toàn bộ config $ git config --list
📍 3 Cấp Độ Config
  • --system: Toàn máy (mọi user) - /etc/gitconfig
  • --global: User hiện tại - ~/.gitconfig
  • --local (mặc định): Repo hiện tại - .git/config
Mức độ ưu tiên: local > global > system.

2.3. SSH Key vs HTTPS

🔐 SSH

  • Authenticate bằng cặp khóa công khai/riêng tư
  • Không cần nhập password mỗi lần push
  • Setup 1 lần dùng mãi
  • URL: git@github.com:user/repo.git
vs

🌐 HTTPS

  • Authenticate bằng username + Personal Access Token
  • Dễ setup, hoạt động qua firewall doanh nghiệp
  • Cần credential helper để cache token
  • URL: https://github.com/user/repo.git

Tạo SSH Key (khuyến nghị)

Khái Niệm Nền Tảng

3.1. Repository (Repo) là gì?

Repository là 1 thư mục chứa source code + toàn bộ lịch sử thay đổi (trong thư mục ẩn .git). Có 2 loại:

  • Local repository: Trên máy bạn
  • Remote repository: Trên server (GitHub, GitLab...)

3.2. 3 Vùng Của Git - Quan Trọng Nhất Phải Hiểu!

┌────────────────────┐ git add ┌────────────────────┐ git commit ┌────────────────────┐ │ │ ──────────→ │ │ ──────────→ │ │ │ Working Directory │ │ Staging Area │ │ Local Repository │ │ (Thư mục làm việc)│ ←────────── │ (Index) │ ←────────── │ (.git folder) │ │ │ git restore │ │ git reset │ │ └────────────────────┘ └────────────────────┘ └────────────────────┘ │ │ git push ↓ ┌────────────────────┐ │ Remote Repository │ │ (GitHub/GitLab) │ └────────────────────┘
📂

Working Directory

Nơi bạn code, sửa file. File ở đây có thể là untracked (chưa biết) hoặc modified (đã sửa).

📋

Staging Area (Index)

"Phòng chờ" trước khi commit. Bạn chọn file nào sẽ commit bằng git add.

📦

Local Repository

Lịch sử chính thức trong .git/. Mỗi git commit lưu 1 snapshot.

3.3. 3 Trạng Thái File

Trạng tháiMô tảVùng
UntrackedFile mới, Git chưa biếtWorking Directory
ModifiedFile đã thay đổi nhưng chưa stagedWorking Directory
StagedĐã thêm vào staging, sẵn sàng commitStaging Area
Committed/UnmodifiedĐã lưu vào lịch sửLocal Repository

3.4. Commit Hash và HEAD

Mỗi commit có 1 SHA-1 hash duy nhất, dài 40 ký tự, ví dụ:

commit a1b2c3d4e5f6789012345678901234567890abcd Author: Nguyen Van A Date: Wed May 20 10:30:00 2026 +0700 feat: thêm chức năng login

Thường chỉ dùng 7 ký tự đầu để gọi commit (a1b2c3d). HEAD là 1 con trỏ đặc biệt - trỏ vào commit hiện tại trong branch hiện tại.

3.5. Snapshot - Cách Git Lưu Dữ Liệu

⚡ Git ≠ Diff-based
Khác với SVN/CVS lưu các bản "delta" (chỉ thay đổi), Git lưu snapshot toàn bộ file tại mỗi commit. Nếu file không thay đổi, Git chỉ tạo 1 reference đến file cũ → vẫn nhanh và tiết kiệm. Đây là lý do Git rất nhanh khi log/diff/branch.

Lệnh Cơ Bản - Workflow Hàng Ngày

4.1. git init - Khởi Tạo Repository

$ mkdir my-project && cd my-project $ git init Initialized empty Git repository in /home/user/my-project/.git/

Lệnh này tạo thư mục ẩn .git chứa toàn bộ metadata. Đừng xóa nó - mất hết lịch sử!

4.2. git clone - Sao Chép Repo Về Máy

# HTTPS $ git clone https://github.com/torvalds/linux.git # SSH $ git clone git@github.com:torvalds/linux.git # Clone vào tên thư mục khác $ git clone https://... my-folder # Clone chỉ 1 branch, depth=1 (siêu nhẹ, dùng cho CI) $ git clone --depth=1 --branch main https://...

4.3. git status - Trạng Thái Hiện Tại

$ git status On branch main Your branch is up to date with 'origin/main'. Changes to be committed: modified: README.md new file: src/login.js Changes not staged for commit: modified: package.json Untracked files: .env.local

Lệnh dùng nhiều nhất trong workflow Git. Hiển thị: branch hiện tại, file staged/modified/untracked.

💡 Tip
Dùng git status -s (short) cho output gọn gàng hơn.

4.4. git add - Đưa File Vào Staging

# Add 1 file $ git add README.md # Add tất cả file thay đổi (kể cả untracked) $ git add . # Add theo pattern $ git add "*.js" # Interactive - chọn từng phần của file (cực mạnh!) $ git add -p

4.5. git commit - Lưu Vào Lịch Sử

# Commit với message $ git commit -m "feat: thêm chức năng login" # Commit + add tất cả modified file (không add untracked) $ git commit -am "fix: sửa bug logout" # Sửa commit gần nhất (chưa push!) $ git commit --amend # Commit không thay đổi message $ git commit --amend --no-edit
⚠️ Cẩn thận với --amend
Lệnh --amend tạo commit MỚI thay thế commit cũ. Không bao giờ amend commit đã push lên remote chung, sẽ phá lịch sử của người khác!

4.6. git log - Xem Lịch Sử

$ git log $ git log --oneline # Gọn, mỗi commit 1 dòng $ git log --graph --oneline --all # Biểu đồ branch ASCII $ git log -n 5 # 5 commit gần nhất $ git log --author="Nguyen" # Lọc theo tác giả $ git log --since="2 weeks ago" $ git log -p file.js # Lịch sử + diff của 1 file

4.7. git diff - So Sánh Thay Đổi

LệnhSo sánh

git diff --stagedStaging vs Last Commit
git diff HEADWorking Dir vs Last Commit
git diff main featureBranch main vs branch feature

4.8. .gitignore - File Bỏ Qua

File .gitignore liệt kê pattern các file/folder Git sẽ không theo dõi.

# .gitignore mẫu cho Node.js node_modules/ dist/ build/ *.log .env .env.local .DS_Store .vscode/ .idea/ *.swp # Bỏ qua tất cả .txt trừ important.txt *.txt !important.txt

Tham khảo github.com/gitignore - bộ sưu tập .gitignore chuẩn cho mọi ngôn ngữ/framework.

Branching - Sức Mạnh Thực Sự Của Git

5.1. Tại Sao Cần Branch?

Branch (nhánh) cho phép bạn phân tách dòng phát triển ra khỏi nhánh chính để:

  • Phát triển tính năng mới mà không ảnh hưởng main
  • Sửa bug khẩn cấp (hotfix) song song với feature đang dở
  • Thử nghiệm ý tưởng - xóa branch là quên
  • Code review trước khi merge vào main (qua Pull Request)
feature/login ↓ ●─────●─────● (HEAD) / main: ●─────●─────●─────●─────●─────●─────● ↑ HEAD trước khi tạo branch

5.2. Lệnh git branch

# Liệt kê branch local $ git branch * main feature/login bugfix/header # Liệt kê tất cả (cả remote) $ git branch -a # Tạo branch mới (KHÔNG chuyển sang) $ git branch feature/payment # Xóa branch (an toàn - chỉ xóa nếu đã merge) $ git branch -d feature/login # Xóa cưỡng bức (CẨN THẬN - mất code chưa merge!) $ git branch -D feature/login # Đổi tên branch hiện tại $ git branch -m new-name

5.3. git checkout vs git switch

Từ Git 2.23 (2019), checkout được tách thành 2 lệnh chuyên biệt: switch (chuyển branch) và restore (khôi phục file).

# Cũ (vẫn hoạt động): $ git checkout feature/login $ git checkout -b new-branch # Tạo + chuyển # Mới (khuyến khích dùng): $ git switch feature/login $ git switch -c new-branch # Create + switch $ git switch - # Quay lại branch trước (như cd -)

5.4. Đặt Tên Branch Theo Convention

PrefixMục đíchVí dụ
feature/Tính năng mớifeature/user-login
bugfix/Sửa bug thườngbugfix/login-redirect
hotfix/Sửa bug khẩn cấp prodhotfix/payment-down
release/Chuẩn bị releaserelease/v2.1.0
chore/Việc lặt vặt (deps, config)chore/update-deps
docs/Tài liệudocs/api-readme

5.5. Detached HEAD - Trạng Thái "Đầu Lìa Khỏi Thân"

Khi bạn git checkout trực tiếp vào 1 commit hash (không phải branch), HEAD trỏ thẳng vào commit đó → detached HEAD. Commit mới sẽ KHÔNG thuộc branch nào → dễ mất.

$ git checkout a1b2c3d You are in 'detached HEAD' state... # Để cứu code, tạo branch ngay: $ git switch -c save-my-work

Merge - Hợp Nhất Nhánh

6.1. Fast-Forward Merge

Xảy ra khi branch đích KHÔNG có commit mới nào kể từ lúc tạo branch nguồn. Git chỉ cần "kéo" con trỏ về phía trước.

TRƯỚC: main: A───B───C ↓ feature: D───E───F (HEAD) SAU `git checkout main && git merge feature`: main: A───B───C───D───E───F (HEAD) ↑ feature

6.2. 3-Way Merge (Recursive)

Khi cả 2 branch đều có commit mới, Git tạo merge commit với 2 parent.

TRƯỚC: main: A───B───C───G───H (HEAD) \ D───E───F (feature) SAU `git merge feature`: main: A───B───C───G───H───M (HEAD) \ / D───E───F / ↑ feature (M = merge commit, có 2 parent: H và F)

6.3. Squash Merge

Gộp toàn bộ commit của branch nguồn thành 1 commit duy nhất trên branch đích. Lịch sử sạch sẽ.

$ git checkout main $ git merge --squash feature/login $ git commit -m "feat: thêm chức năng login (squashed)"

6.4. So Sánh 3 Loại Merge

LoạiKhi nào dùngƯuNhược
Fast-forwardBranch nguồn ngắn, không cần lưu lạiLịch sử thẳng, đơn giảnMất thông tin "đây là 1 feature"
3-way (--no-ff)Feature lớn, cần thấy được nhánhBảo tồn cấu trúc branchLịch sử rối, nhiều merge commit
SquashFeature nhiều commit lẻ tẻ (WIP, fix typo)Sạch, dễ đọc, dễ revertMất chi tiết quá trình phát triển

6.5. --no-ff: Bắt Buộc Tạo Merge Commit

$ git merge --no-ff feature/login

Dù có thể fast-forward, vẫn tạo merge commit để đánh dấu "đây là chỗ feature được merge vào". Nhiều team dùng cho main branch.

Remote & Cộng Tác Nhóm

7.1. Quản Lý Remote

# Liệt kê remote $ git remote -v origin git@github.com:user/repo.git (fetch) origin git@github.com:user/repo.git (push) # Thêm remote mới $ git remote add upstream git@github.com:original/repo.git # Đổi URL $ git remote set-url origin git@github.com:new/repo.git # Xóa $ git remote remove upstream

7.2. fetch vs pull - KHÁC NHAU!

📥 git fetch

  • Tải code mới từ remote về
  • KHÔNG tự động merge vào branch hiện tại
  • An toàn - không thay đổi working dir
  • Cập nhật các tracking branch (origin/main)
$ git fetch origin
vs

📥+🔀 git pull

  • = fetch + merge
  • Tải về VÀ merge vào branch hiện tại
  • Có thể gây conflict
  • --rebase để rebase thay vì merge
$ git pull --rebase

7.3. git push

# Push branch hiện tại lên origin $ git push # Lần đầu push branch mới: cần set upstream $ git push -u origin feature/login # Push tất cả branch $ git push --all # Push tag $ git push --tags # Force push (NGUY HIỂM) $ git push --force-with-lease # An toàn hơn --force

7.4. Origin vs Upstream - Forking Workflow

Khi fork 1 repo trên GitHub, convention là:

  • origin → fork của BẠN (bạn có quyền push)
  • upstream → repo GỐC (chỉ pull để sync)
# Đồng bộ fork với upstream $ git fetch upstream $ git checkout main $ git merge upstream/main $ git push origin main

7.5. Force Push - Khi Nào & Cẩn Thận!

🚨 Quy tắc Force Push
  • Không bao giờ force push lên main hay develop
  • Chỉ force push lên branch CỦA RIÊNG BẠN (sau rebase/amend)
  • Luôn dùng --force-with-lease thay vì --force - an toàn hơn vì check remote chưa thay đổi
  • Trao đổi với team trước khi force push branch chia sẻ

7.6. Pull Request / Merge Request Workflow

1

git switch -c feature/login
2

Commit nhỏ, message rõ ràng theo Conventional Commits

Xử Lý Conflict - Nỗi Ám Ảnh Của Developer

8.1. Tại Sao Có Conflict?

Conflict xảy ra khi Git không thể tự động merge 2 branch vì cả hai cùng sửa cùng 1 dòng trong cùng 1 file theo cách khác nhau.

8.2. Cấu Trúc Conflict Markers

function login(user) { <<<<<<< HEAD ← Phần code BRANCH HIỆN TẠI (main) return authenticate(user.email, user.password); ======= ← Phân cách return authService.login(user); >>>>>>> feature/oauth ← Phần code BRANCH ĐANG MERGE VÀO }

8.3. Quy Trình Giải Quyết Conflict

1

Chạy git status để xem file conflict

File hiện trong "Unmerged paths"
2

Mở file, đọc kỹ cả 2 phía

Hiểu logic cả 2 thay đổi - hỏi đồng nghiệp nếu cần
3

Sửa code, xóa markers

Giữ phần đúng, xóa <<<<, ====, >>>>
4

git add file đã sửa

Đánh dấu conflict đã giải quyết
5

git commit để hoàn tất merge

Git tự sinh message merge

8.4. Tools Hỗ Trợ Giải Quyết

VS Code

Nhận diện tự động, nút "Accept Current/Incoming/Both"

git mergetool

Mở merge tool đã config (kdiff3, vimdiff, meld)

GitKraken/Fork

3-way merge editor trực quan, side-by-side

8.5. Hủy Merge Giữa Chừng

# Quay lại trạng thái trước khi bắt đầu merge $ git merge --abort # Tương tự cho rebase $ git rebase --abort # Hoặc cherry-pick $ git cherry-pick --abort

8.6. Chiến Lược Tránh Conflict

  • Pull thường xuyên: Sync với main mỗi ngày, conflict nhỏ dễ giải quyết
  • Branch ngắn hạn: Feature branch không quá 1-2 tuần
  • Phân chia file: Mỗi người làm module riêng, ít đụng nhau
  • Code style chuẩn: Prettier/ESLint tự động format → tránh conflict do tab/space
  • Communicate: Báo cho team biết bạn đang refactor file lớn

Undo - Hoàn Tác Mọi Thứ

9.1. git restore (Git 2.23+)

# Khôi phục file working dir về trạng thái như staging $ git restore file.js # Khôi phục về trạng thái như HEAD (xóa cả thay đổi đã stage) $ git restore --source=HEAD --staged --worktree file.js # Unstage file (đẩy ra khỏi staging) $ git restore --staged file.js

9.2. git reset - 3 Mức Độ

LệnhWorking DirStagingHEADDùng khi
--softGiữ nguyênGiữ nguyênLùiĐổi message commit cuối, gộp commit
--mixed (mặc định)Giữ nguyênResetLùiUnstage tất cả, làm lại từ add
--hard ⚠️ResetResetLùiVứt sạch mọi thứ về commit cũ
# Hủy commit cuối, giữ code đã sửa trong working dir $ git reset --soft HEAD~1 # Hủy 3 commit gần nhất, giữ code $ git reset --mixed HEAD~3 # NGUY HIỂM: Reset cứng về commit cụ thể - mất hết thay đổi sau đó $ git reset --hard abc1234

9.3. git revert - An Toàn Hơn Reset

Tạo 1 commit MỚI để hoàn tác tác động của commit cũ. Không thay đổi lịch sử → an toàn cho branch chia sẻ.

$ git revert abc1234 # Mở editor để viết message, sau đó tạo commit ngược # Revert nhiều commit liên tiếp $ git revert HEAD~3..HEAD # Revert nhưng không tự commit (để gộp nhiều revert) $ git revert --no-commit abc1234 def5678

9.4. So Sánh Reset vs Revert vs Restore

LệnhThay đổi lịch sử?An toàn cho remote?Phạm vi
git restoreKhôngCóFile trong WD/Staging
git resetCóKhông (nếu đã push)Commit lùi HEAD
git revertKhôngCóTạo commit ngược

9.5. git reflog - Lá Bùa Cứu Mạng!

Reflog ghi lại MỌI thay đổi của HEAD trong 90 ngày. Khi bạn nghĩ "tôi mất commit rồi!", reflog gần như luôn cứu được.

$ git reflog a1b2c3d HEAD@{0}: reset: moving to HEAD~5 e4f5g6h HEAD@{1}: commit: feat: add payment i7j8k9l HEAD@{2}: commit: fix: login bug ... # "Cứu" commit đã reset --hard mất $ git reset --hard HEAD@{1}
🛡️ Nhớ kỹ
Nếu lỡ git reset --hard mất code, ĐỪNG hoảng. Mở reflog ngay - 99% trường hợp vẫn cứu được.

9.6. git clean - Dọn File Untracked

# Dry run - xem sẽ xóa gì (LUÔN chạy trước) $ git clean -n # Xóa file untracked $ git clean -f # Xóa cả thư mục $ git clean -fd # Xóa cả file trong .gitignore (RESET HOÀN TOÀN) $ git clean -fdx

Stash & Tag

10.1. git stash - Cất Tạm Code

Bạn đang code dở thì sếp bảo "fix gấp bug prod". Bạn không muốn commit code dở nhưng cũng không thể chuyển branch (Git không cho). → git stash giải quyết!

# Cất tạm $ git stash # Cất kèm message (KHUYẾN KHÍCH) $ git stash push -m "WIP: chức năng login dang dở" # Cất cả file untracked $ git stash -u # Liệt kê $ git stash list stash@{0}: WIP: chức năng login dang dở stash@{1}: WIP on main: fix typo # Lấy ra (và xóa stash) $ git stash pop # Lấy ra nhưng GIỮ stash (apply) $ git stash apply stash@{1} # Xóa stash $ git stash drop stash@{0} # Xóa hết stash $ git stash clear

10.2. git tag - Đánh Dấu Phiên Bản

Tag dùng để đánh dấu phiên bản release (v1.0.0, v2.1.3...).

Lightweight Tag vs Annotated Tag

🏷️ Lightweight

  • Chỉ là 1 con trỏ đến commit
  • Không có author, date, message
  • Như "bookmark" tạm
$ git tag v1.0
vs

🎖️ Annotated (khuyến nghị)

  • Object hoàn chỉnh trong Git
  • Có author, date, message, GPG sign
  • Dùng cho production release
$ git tag -a v1.0.0 -m "Release 1.0.0"
# Liệt kê tag $ git tag # Tag 1 commit cũ $ git tag -a v0.9 abc1234 -m "Beta release" # Push tag lên remote (push KHÔNG tự push tag!) $ git push origin v1.0.0 $ git push --tags # Push tất cả tag # Xóa tag $ git tag -d v1.0.0 # Local $ git push origin --delete v1.0.0 # Remote

10.3. Semantic Versioning (SemVer)

📐 Format: MAJOR.MINOR.PATCH
  • MAJOR: Breaking changes (không tương thích ngược) - v1.x.x → v2.0.0
  • MINOR: Tính năng mới (vẫn tương thích) - v1.2.x → v1.3.0
  • PATCH: Bug fix - v1.2.3 → v1.2.4
  • Pre-release: v1.0.0-alpha.1, v1.0.0-rc.1

Rebase - Nghệ Thuật Viết Lại Lịch Sử

11.1. Rebase vs Merge - Bản Chất Khác Nhau

🔀 Merge

  • Tạo merge commit, giữ cấu trúc branch
  • Bảo tồn lịch sử chính xác
  • Lịch sử "rối" (đan xen)
vs

📐 Rebase

  • Di chuyển commit sang đỉnh branch khác
  • Viết lại lịch sử (tạo commit mới với hash mới)
  • Lịch sử thẳng, sạch sẽ
TRƯỚC khi rebase feature lên main: main: A───B───C───G───H \ D───E───F (feature) SAU `git checkout feature && git rebase main`: main: A───B───C───G───H \ D'──E'──F' (feature) (D', E', F' là commit MỚI với nội dung giống D, E, F nhưng hash khác)

11.2. Lệnh git rebase

# Rebase branch hiện tại lên main $ git checkout feature $ git rebase main # Nếu conflict: sửa, add, rồi continue $ git rebase --continue # Bỏ qua commit gây conflict $ git rebase --skip # Hủy rebase, quay về trước $ git rebase --abort

11.3. Interactive Rebase - Siêu Năng Lực

Cho phép bạn chỉnh sửa lịch sử commit: gộp, đổi tên, xóa, đổi thứ tự.

# Sửa 5 commit gần nhất $ git rebase -i HEAD~5

Git sẽ mở editor với nội dung:

pick a1b2c3d feat: add login form pick e4f5g6h fix typo pick i7j8k9l fix another typo pick m1n2o3p style: format code pick q4r5s6t feat: add validation # Commands: # p, pick = use commit # r, reword = use commit, but edit message # e, edit = use commit, but stop for amending # s, squash = use commit, but meld into previous # f, fixup = like "squash", but discard message # d, drop = remove commit

11.4. 6 Lệnh Quan Trọng Trong Interactive Rebase

LệnhTác dụng
pick (p)Giữ nguyên commit
reword (r)Giữ commit nhưng sửa message
edit (e)Dừng lại để bạn amend commit
squash (s)Gộp vào commit phía TRÊN, gộp message
fixup (f)Gộp vào commit phía TRÊN, BỎ message
drop (d)Xóa commit hẳn

11.5. The Golden Rule of Rebasing - Quy Tắc Vàng

🚨 KHÔNG BAO GIỜ rebase branch đã public/shared
Rebase tạo commit mới với hash mới. Nếu branch đã push và đồng nghiệp đã pull, rebase + force push sẽ phá lịch sử của họ. Chỉ rebase branch CỦA RIÊNG BẠN, hoặc khi cả team đồng ý.

11.6. git rebase --onto - Move Commit Tinh Vi

Di chuyển 1 dãy commit từ branch này sang branch khác.

# Lấy commit của feature (sau khi tách từ wrong-branch) và đặt lên main $ git rebase --onto main wrong-branch feature

Cherry-pick & Bisect

12.1. git cherry-pick - "Hái" Commit

Áp dụng 1 (hoặc nhiều) commit cụ thể từ branch khác vào branch hiện tại.

# Pick 1 commit $ git cherry-pick a1b2c3d # Pick nhiều commit $ git cherry-pick a1b2c3d e4f5g6h # Pick range (KHÔNG bao gồm commit đầu) $ git cherry-pick a1b2c3d..e4f5g6h # Pick range (bao gồm cả 2 đầu) $ git cherry-pick a1b2c3d^..e4f5g6h # Pick mà không tự commit (để gộp) $ git cherry-pick --no-commit a1b2c3d
💼 Use case thực tế
Bạn fix bug trên branch develop, nhưng cần áp dụng ngay lên main (hotfix). Thay vì merge cả branch develop (kéo theo tính năng dở), chỉ cherry-pick commit fix.

12.2. git bisect - Truy Tìm Commit Gây Bug

Dùng binary search trong lịch sử để tìm chính xác commit nào đã introduce 1 bug. Cực mạnh khi bug "lén" lọt vào từ lâu.

# Bắt đầu $ git bisect start # Đánh dấu commit hiện tại là BUG $ git bisect bad # Đánh dấu 1 commit cũ là GOOD (chưa có bug) $ git bisect good a1b2c3d # Git tự checkout vào commit giữa - bạn test: # Nếu có bug: git bisect bad # Nếu không bug: git bisect good # Lặp đến khi tìm thấy → Git báo commit "first bad commit" # Kết thúc $ git bisect reset

Bisect Tự Động

# Nếu có script test (exit 0 = good, exit 1 = bad) $ git bisect start HEAD v1.0 $ git bisect run npm test # Git tự checkout, chạy test, đánh dấu, lặp đến khi tìm ra!
📊 Hiệu quả
Với 1000 commit, bisect tìm ra commit lỗi sau ~10 lần test (log₂ 1000 ≈ 10). Linear search tệ nhất cần 1000 lần.

Git Internals - Bên Trong Git

13.1. Thư Mục .git/ - Trái Tim Của Git

$ ls -la .git/ HEAD ← Con trỏ tới branch hiện tại config ← Cấu hình local description ← (chỉ dùng cho GitWeb) hooks/ ← Scripts tự động chạy theo event info/ ← exclude (ignore không cần commit .gitignore) objects/ ← Toàn bộ data: blob, tree, commit, tag refs/ ← Pointers: branches, tags logs/ ← Reflog index ← Staging area

13.2. 4 Loại Git Objects

📄

Blob

Lưu nội dung 1 file (không kèm tên/path). Mỗi nội dung khác nhau = 1 blob khác nhau.

🌳

Tree

Như "thư mục" - chứa tên file, mode, và pointer đến blob/tree con.

📌

Commit

Snapshot của project - chứa: tree gốc, parent commit(s), author, message.

🏷️

Tag (annotated)

Pointer đến 1 object (thường là commit) kèm message + GPG sign.

Commit ─────────┐ │ │ │ tree: ↓ └────────→ Tree (root) │ ├── README.md → Blob ("# My Project...") │ ├── src/ → Tree │ │ ├── main.js → Blob ("console.log...") │ │ └── utils.js → Blob ("...") │ └── docs/ → Tree │ └── api.md → Blob ("...")

13.3. Plumbing vs Porcelain Commands

Porcelain (cao cấp)Plumbing (thấp cấp)
git commitgit hash-object, git update-index, git write-tree, git commit-tree
git checkoutgit read-tree, git update-ref
git loggit rev-list, git diff-tree

Porcelain: lệnh thường dùng (UI-friendly). Plumbing: lệnh nội bộ, dùng cho scripting/automation.

13.4. Khám Phá Object Bằng Lệnh

# Hash của file (chưa cần add) $ echo "hello" | git hash-object --stdin ce013625030ba8dba906f756967f9e9ca394464a # Xem nội dung object $ git cat-file -p ce0136 hello # Xem TYPE của object $ git cat-file -t ce0136 blob

13.5. git gc - Dọn Dẹp

Garbage Collection: nén objects loose thành pack files, xóa object không reference (orphan).

$ git gc $ git gc --aggressive # Nén tốt hơn, lâu hơn $ git gc --prune=now # Xóa cả object dưới 2 tuần

Hooks & Automation

14.1. Git Hooks là gì?

Scripts tự động chạy trước/sau các sự kiện Git (commit, push, merge...). Nằm trong .git/hooks/.

14.2. Client-side Hooks Quan Trọng

HookChạy khiUse case
pre-commitTrước khi commitLint, format, run unit tests
prepare-commit-msgTrước editor mở để viết messageTự sinh template message
commit-msgSau khi viết messageValidate format (Conventional Commits)
post-commitSau commit thành côngNotification
pre-pushTrước khi pushRun integration tests
post-mergeSau merge thành côngReinstall dependencies

14.3. Server-side Hooks

  • pre-receive: Reject push không đúng convention
  • update: Per-branch validation
  • post-receive: Deploy, notify CI/CD

14.4. Ví Dụ: pre-commit Hook

# File: .git/hooks/pre-commit (chmod +x) #!/bin/sh npm run lint if [ $? -ne 0 ]; then echo "❌ Lint failed. Commit bị hủy." exit 1 fi npm test

14.5. Husky + lint-staged - Modern Hook Management

Hook nằm trong .git/hooks/ không được commit lên Git. Husky giải quyết bằng cách lưu hook vào .husky/ và auto-install.

$ npm install -D husky lint-staged $ npx husky init # Thêm pre-commit hook $ echo "npx lint-staged" > .husky/pre-commit
# package.json { "lint-staged": { "*.{js,ts}": ["eslint --fix", "prettier --write"], "*.{css,md}": ["prettier --write"] } }

14.6. Conventional Commits + commitlint

📝 Conventional Commits Format
<type>(<scope>): <subject>
Types: feat, fix, docs, style, refactor, perf, test, build, ci, chore, revert
Ví dụ: feat(auth): thêm OAuth Google login
$ npm install -D @commitlint/cli @commitlint/config-conventional $ echo "module.exports = {extends: ['@commitlint/config-conventional']}" > commitlint.config.js # .husky/commit-msg npx --no -- commitlint --edit $1

Workflows - Chiến Lược Quản Lý Nhánh

15.1. Gitflow (Vincent Driessen, 2010)

Workflow cổ điển với 5 loại branch. Phù hợp cho dự án có release theo lịch (mobile app, desktop software).

BranchMục đíchTạo từMerge vào
mainProduction code--
developIntegration của các featuremainmain (qua release)
feature/*Tính năng mớidevelopdevelop
release/*Chuẩn bị release (test, bug)developmain + develop
hotfix/*Fix bug khẩn cấp productionmainmain + develop

15.2. GitHub Flow - Đơn Giản Hơn

Phù hợp continuous deployment (web app, SaaS).

  1. Tạo branch từ main
  2. Commit, push thường xuyên
  3. Mở Pull Request sớm (kể cả khi chưa xong - WIP)
  4. Review + CI tests
  5. Merge vào main → Deploy ngay

15.3. Trunk-Based Development

Mọi người commit thẳng vào main (trunk) - feature branch cực ngắn (< 1 ngày). Dùng feature flags để ẩn code chưa xong.

🏢 Ai dùng?
Google, Facebook, Microsoft dùng trunk-based. Yêu cầu: CI/CD mạnh, test automation tốt, feature flags.

📱 App mobile có release cycle

→ Gitflow. Cần hotfix branch song song với feature đang dở.

🌐 SaaS deploy liên tục

→ GitHub Flow. Đơn giản, nhanh, dễ rollback.

🏭 Microservices ở big tech

→ Trunk-based + feature flags. Tối đa hóa tốc độ.

🌍 Open-source project

→ Forking workflow. Contributor fork, owner merge PR.

Tips, Best Practices & Troubleshooting

16.1. Commit Message Chuẩn (Conventional Commits)

# ✅ TỐT feat(auth): thêm Google OAuth login fix(api): timeout khi gọi /users với pagination > 100 docs: cập nhật README phần cài đặt refactor(user): tách logic validation ra module riêng # ❌ TỆ update fix bug WIP asdf quick fix

16.2. Git Aliases - Tăng Tốc Làm Việc

$ git config --global alias.co checkout $ git config --global alias.br branch $ git config --global alias.st status $ git config --global alias.cm "commit -m" $ git config --global alias.lg "log --oneline --graph --all --decorate" $ git config --global alias.amend "commit --amend --no-edit" $ git config --global alias.undo "reset HEAD~1 --soft"

16.3. Git LFS - Quản Lý File Lớn

Git xử lý file binary lớn (video, PSD, dataset) rất kém - mỗi version copy toàn bộ. Git LFS (Large File Storage) lưu file lớn ở server riêng, repo chỉ chứa pointer.

$ git lfs install $ git lfs track "*.psd" $ git lfs track "*.mp4" $ git add .gitattributes $ git add file.psd $ git commit -m "add design"

16.4. Submodules & Worktree

Submodule - Repo Trong Repo

$ git submodule add https://github.com/lib/dep.git vendor/dep $ git submodule update --init --recursive

Worktree - Nhiều Working Dir Song Song

# Tạo working dir cho branch khác mà không phải clone lại $ git worktree add ../my-feature feature/login $ cd ../my-feature # Giờ bạn có 2 thư mục, mỗi cái 1 branch khác nhau! $ git worktree list $ git worktree remove ../my-feature

16.5. GPG Sign Commits - Chữ Ký Số

Chứng minh commit thật sự do BẠN tạo (không phải ai mạo danh bằng user.email).

$ gpg --gen-key $ git config --global user.signingkey YOUR_KEY_ID $ git config --global commit.gpgsign true # Mỗi commit sẽ có badge "Verified" trên GitHub

16.6. Recovery Scenarios - Cứu Mạng

💀 "Tôi lỡ git reset --hard, mất hết code!"

$ git reflog $ git reset --hard HEAD@{1}

💀 "Tôi commit lên main thay vì feature branch!"

# Đang ở main, đã commit nhầm $ git branch feature/save-work # Tạo branch giữ commit $ git reset --hard HEAD~1 # Lùi main về trước commit $ git checkout feature/save-work # Sang branch mới

💀 "Tôi push code có password lên GitHub!"

🚨 Khẩn cấp
  1. Đổi password/token ngay - coi như đã lộ
  2. Dùng git filter-repo hoặc BFG Repo-Cleaner xóa khỏi lịch sử
  3. Force push lên remote (đã thông báo team)
  4. GitHub có thể vẫn cache - cần liên hệ support

💀 "Conflict trong rebase, sợ quá!"

$ git rebase --abort # Hủy, quay về như chưa rebase

16.7. Performance Tips Cho Repo Lớn

  • Partial clone: git clone --filter=blob:none
  • Shallow clone: git clone --depth=1 (chỉ commit gần nhất)
  • Sparse checkout: Chỉ checkout 1 phần của monorepo
  • git maintenance: git maintenance start (Git 2.30+)
  • fsmonitor: Theo dõi file change bằng OS-native API
