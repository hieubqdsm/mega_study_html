CH 1 Design pattern là gì & KHÔNG là gì?

Hiểu đúng bản chất trước khi đi vào từng pattern.

Một design pattern là khuôn mẫu giải pháp cho một vấn đề thiết kế thường gặp — không phải đoạn code copy-paste, mà là cách tổ chức các lớp/đối tượng và quan hệ giữa chúng. Cuốn kinh điển "Design Patterns" (1994) của nhóm Gang of Four (GoF) mô tả 23 pattern, đến nay vẫn là nền tảng.

Pattern LÀ gì

  • Từ vựng chung: "ở đây dùng Adapter" — ai cũng hiểu ngay cấu trúc.
  • Giải pháp đã kiểm chứng: tránh tự mò ra cách dở.
  • Độc lập ngôn ngữ: ý tưởng giống nhau ở PHP, C, Python, Java...

Pattern KHÔNG phải là gì

  • Không phải thư viện hay framework để cài đặt — nó là ý tưởng thiết kế.
  • Không phải "càng nhiều càng tốt" — lạm dụng pattern (over-engineering) làm code rối hơn.
  • Không phải luật cứng — đôi khi một if/else đơn giản tốt hơn cả một Strategy.
Sai lầm người mớiHọc xong hay có xu hướng "nhét pattern vào mọi chỗ". Pattern chỉ đáng dùng khi vấn đề nó giải quyết thực sự xuất hiện (code đang khó mở rộng, lặp lại, khó test). Nhận diện khi nào KHÔNG cần cũng quan trọng như biết pattern.
SOLID — nền của hầu hết patternĐa số pattern là cách hiện thực các nguyên tắc SOLID, đặc biệt: O (Open/Closed — mở để mở rộng, đóng để sửa) và D (Dependency Inversion — phụ thuộc vào abstraction, không vào implementation). Khi thấy code "lập trình theo interface, không theo lớp cụ thể" — đó là dấu hiệu có pattern.

CH 2 Ba nhóm pattern & cách đọc sơ đồ

23 pattern GoF chia làm 3 nhóm theo mục đích. Biết nhóm giúp khoanh vùng khi nhận diện.

NhómTrả lời câu hỏiGồm các pattern
Creational
(Khởi tạo)		"Tạo đối tượng như thế nào?"Singleton, Factory Method, Abstract Factory, Builder, Prototype
Structural
(Cấu trúc)		"Ghép các đối tượng thành cấu trúc lớn ra sao?"Adapter, Decorator, Facade, Composite, Proxy, Bridge, Flyweight

Mẹo khoanh vùng nhanh khi nhận diện

🧭 Hỏi 3 câu để biết thuộc nhóm nào
  • Code đang bận tâm việc tạo / new đối tượng (giấu new, tái dùng instance)? → Creational.
  • Code đang bọc / ghép / chuyển đổi một đối tượng để nó hợp với chỗ khác? → Structural.
  • Code đang bận tâm luồng xử lý, ai gọi ai, thuật toán nào chạy? → Behavioral.
Đọc sơ đồ UML tối giảnTrong khóa này: A ──▷ B nghĩa là "A kế thừa / hiện thực B"; A ──> B nghĩa là "A giữ tham chiếu tới B (composition)". Hầu hết pattern = một interface + vài lớp hiện thực + một lớp "client" giữ tham chiếu qua interface. Nhận ra bộ ba này là nhận ra 80% pattern.

CH 3 Quy trình nhận diện một pattern

Checklist 4 bước áp dụng cho bất kỳ đoạn code lạ nào.

🔬 Checklist nhận diện
  • Bước 1 — Tên gọi: tên class/file/method có "rò rỉ" pattern không? XxxFactory, XxxBuilder, XxxStrategy, XxxObserver, getInstance, subscribe/notify...
  • Bước 2 — Cấu trúc: có một interface/abstract + nhiều lớp con hoán đổi được không? Có lớp "bọc" lớp khác cùng interface không?
  • Bước 3 — Ý đồ: code đang cố đạt điều gì? (giấu việc khởi tạo? thêm hành vi mà không sửa lớp gốc? cho phép đổi thuật toán lúc chạy?)
  • Bước 4 — Đối chiếu: so với "Dấu hiệu nhận biết" của từng pattern trong khóa để chốt.
Quy tắc ngón tay cáiNhiều pattern trông giống nhau về cấu trúc (Strategy, State, Command, Adapter, Decorator đều là "một lớp giữ tham chiếu tới interface"). Phân biệt chúng bằng Ý ĐỒ, không bằng sơ đồ. Ví dụ: Strategy & State cấu trúc gần như y hệt — khác ở mục đích (đổi thuật toán vs. đổi trạng thái).
Sẵn sàng vào patternTừ đây mỗi chương là một pattern. Nhớ: đọc Dấu hiệu nhận biết kỹ nhất — đó là phần luyện "con mắt" của bạn.
CR · Khởi tạo (Creational)
Nhóm CreationalTất cả nói về cách tạo đối tượng: giấu toán tử new, kiểm soát số lượng instance, hoặc dựng object phức tạp từng bước. Dấu hiệu chung: code không gọi new trực tiếp mà thông qua một method/đối tượng trung gian.

CH 4 Singleton Creational

Đảm bảo một lớp chỉ có đúng một instance, và cung cấp điểm truy cập toàn cục tới nó.

🎯 Ý đồ:"Chỉ được tồn tại một". Dùng cho thứ vốn dĩ chỉ nên có một: connection pool, config, logger, cache.
🔎 Dấu hiệu nhận biết Singleton
  • Có method tĩnh getInstance() / instance() / shared().
  • Một biến static giữ instance, khởi tạo lười (chỉ tạo lần đầu).
  • Constructor bị private (PHP/Java) hoặc __new__ bị ghi đè (Python) — chặn new từ ngoài.
  • Mọi nơi trong code gọi X::getInstance() thay vì new X().
<?php
class Database {
    private static ?Database $instance = null;   // ô nhớ giữ instance duy nhất
    private function __construct() {}             // chặn `new Database()` từ bên ngoài

    public static function getInstance(): Database {
        if (self::$instance === null) {          // lần đầu mới tạo (lazy)
            self::$instance = new Database();
        }
        return self::$instance;                  // các lần sau trả lại y chang
    }
    public function query(string $sql) { /* ... */ }
}

$a = Database::getInstance();
$b = Database::getInstance();
// $a === $b  -> TRUE, cùng một đối tượng
// C không có class -> dùng hàm + biến static trong hàm
typedef struct { int conn; } Database;

Database *db_instance(void) {
    static Database inst;            // sống suốt vòng đời chương trình
    static int ready = 0;
    if (!ready) {                    // khởi tạo lười, đúng 1 lần
        inst.conn = open_connection();
        ready = 1;
    }
    return &inst;                    // mọi lời gọi trả CÙNG địa chỉ
}

Database *a = db_instance();
Database *b = db_instance();
// a == b  -> cùng con trỏ
class Database:
    _instance = None                      # biến lớp giữ instance

    def __new__(cls):                     # ghi đè bước tạo object
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance

    def query(self, sql): ...

a = Database()
b = Database()
print(a is b)   # True - cùng một object
Mặt trái — vì sao nhiều người ghét SingletonNó là biến toàn cục trá hình: gây phụ thuộc ngầm, khó test (khó thay mock), khó chạy song song (đa luồng cần khóa). Nay người ta thường thay bằng Dependency Injection (truyền instance vào thay vì gọi getInstance()). Nhận diện được nó để biết "ồ chỗ này khó test vì Singleton".
Nhầm với...Một class chỉ tình cờ được tạo 1 lần không phải Singleton — Singleton là khi lớp tự ép buộc chỉ có một. Khác Factory: Factory tạo ra nhiều đối tượng mới; Singleton tái dùng một.

CH 5 Factory Method Creational

Định nghĩa một method để tạo đối tượng, nhưng để lớp con quyết định tạo lớp cụ thể nào.

🎯 Ý đồ:Giấu việc new đằng sau một method, để thêm loại sản phẩm mới mà không sửa code gọi.
🔎 Dấu hiệu nhận biết Factory Method
  • Có method tên create*() / make*() trả về một interface/abstract, không trả lớp cụ thể.
  • Bên trong là một switch/if trên "kiểu" để quyết định new lớp nào.
  • Code client chỉ làm việc với kiểu trừu tượng (Shape), không biết lớp thật (Circle).
  • Thêm sản phẩm mới = thêm một nhánh / một lớp con, không đụng client.
<?php
interface Shape { public function area(): float; }
class Circle implements Shape { /*...*/ public function area(): float { return 3.14; } }
class Square implements Shape { /*...*/ public function area(): float { return 4.0; } }

class ShapeFactory {
    public static function create(string $type): Shape {   // FACTORY METHOD
        return match($type) {
            'circle' => new Circle(),
            'square' => new Square(),
            default  => throw new InvalidArgumentException($type),
        };
    }
}
$s = ShapeFactory::create('circle');   // client KHÔNG gọi `new Circle`
typedef struct { float (*area)(void); } Shape;   // "interface" = struct con trỏ hàm

static float circle_area(void){ return 3.14f; }
static float square_area(void){ return 4.0f;  }

Shape make_shape(const char *type) {             // FACTORY METHOD
    Shape s;
    if (strcmp(type,"circle")==0) s.area = circle_area;
    else                          s.area = square_area;
    return s;
}
Shape s = make_shape("circle");
float a = s.area();   // gọi qua con trỏ hàm, không biết loại cụ thể
from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self): ...

class Circle(Shape):
    def area(self): return 3.14
class Square(Shape):
    def area(self): return 4.0

def make_shape(kind: str) -> Shape:      # FACTORY METHOD
    return {'circle': Circle, 'square': Square}[kind]()

s = make_shape('circle')   # client không gọi Circle() trực tiếp
Nhầm với Abstract FactoryFactory Method tạo một loại sản phẩm. Abstract Factory (CH6) tạo cả một bộ sản phẩm liên quan. Nếu chỉ thấy một create() trả một interface → Factory Method.

CH 6 Abstract Factory Creational

Tạo ra các họ đối tượng liên quan mà không chỉ rõ lớp cụ thể — đảm bảo chúng "ăn khớp" nhau.

🎯 Ý đồ:Một factory tạo nhiều sản phẩm cùng "phong cách" (vd toàn bộ UI theo theme Dark, hoặc theme Light), không lẫn lộn.
🔎 Dấu hiệu nhận biết Abstract Factory
  • Một interface factory có nhiều method create: createButton(), createCheckbox()...
  • Có nhiều factory cụ thể (DarkFactory, LightFactory), mỗi cái tạo ra một bộ sản phẩm khớp nhau.
  • Client nhận một factory rồi gọi các create — không bao giờ trộn sản phẩm của 2 họ.
<?php
interface Button   { public function render(): string; }
interface Checkbox { public function render(): string; }

interface GUIFactory {                       // ABSTRACT FACTORY
    public function createButton(): Button;
    public function createCheckbox(): Checkbox;
}
class DarkFactory implements GUIFactory {    // một "họ" sản phẩm
    public function createButton(): Button   { return new DarkButton(); }
    public function createCheckbox(): Checkbox{ return new DarkCheckbox(); }
}
// LightFactory tương tự -> toàn bộ UI nhất quán theo theme
function buildUI(GUIFactory $f) {
    echo $f->createButton()->render();
    echo $f->createCheckbox()->render();
}
typedef struct { char* (*render)(void); } Button;
typedef struct { char* (*render)(void); } Checkbox;

typedef struct {                         // ABSTRACT FACTORY = struct các con trỏ hàm
    Button   (*create_button)(void);
    Checkbox (*create_checkbox)(void);
} GUIFactory;

static Button dark_button(void){ Button b={dark_btn_render}; return b; }
static Checkbox dark_checkbox(void){ Checkbox c={dark_chk_render}; return c; }

GUIFactory dark_factory(void){            // một họ sản phẩm
    GUIFactory f = { dark_button, dark_checkbox };
    return f;
}
void build_ui(GUIFactory f){ f.create_button(); f.create_checkbox(); }
from abc import ABC, abstractmethod

class GUIFactory(ABC):                    # ABSTRACT FACTORY
    @abstractmethod
    def create_button(self): ...
    @abstractmethod
    def create_checkbox(self): ...

class DarkFactory(GUIFactory):            # một họ sản phẩm nhất quán
    def create_button(self):   return DarkButton()
    def create_checkbox(self): return DarkCheckbox()

def build_ui(factory: GUIFactory):
    factory.create_button().render()
    factory.create_checkbox().render()
Phân biệt nhanhThấy một method create → Factory Method. Thấy nhiều method create gom trong một interface, và có nhiều "họ" → Abstract Factory.

CH 7 Builder Creational

Dựng một đối tượng phức tạp từng bước, tách quá trình xây khỏi biểu diễn cuối.

🎯 Ý đồ:Tránh constructor "khủng long" 15 tham số. Cho phép cấu hình dần rồi build().
🔎 Dấu hiệu nhận biết Builder
  • Chuỗi method trả về $this/self để gọi nối nhau (fluent / method chaining): ->setX()->setY()->build().
  • Một method build() / getResult() ở cuối trả ra sản phẩm hoàn chỉnh.
  • Đối tượng có nhiều phần tùy chọn; thay cho constructor quá nhiều tham số.
<?php
class PizzaBuilder {
    private array $toppings = [];
    private string $size = 'M';
    public function size(string $s): static { $this->size = $s; return $this; }  // return $this -> nối chuỗi
    public function add(string $t): static   { $this->toppings[] = $t; return $this; }
    public function build(): Pizza           { return new Pizza($this->size, $this->toppings); }
}
$pizza = (new PizzaBuilder())
    ->size('L')->add('cheese')->add('mushroom')   // fluent
    ->build();
typedef struct { char size; const char* toppings[8]; int n; } Pizza;

typedef struct { Pizza p; } PizzaBuilder;
PizzaBuilder* pb_size(PizzaBuilder* b, char s){ b->p.size = s; return b; }   // trả con trỏ -> nối chuỗi
PizzaBuilder* pb_add (PizzaBuilder* b, const char* t){ b->p.toppings[b->p.n++]=t; return b; }
Pizza pb_build(PizzaBuilder* b){ return b->p; }

PizzaBuilder b = {0};
Pizza pizza = pb_build( pb_add( pb_add( pb_size(&b,'L'), "cheese"), "mushroom") );
class PizzaBuilder:
    def __init__(self):
        self.size, self.toppings = 'M', []
    def set_size(self, s):                 # trả self -> method chaining
        self.size = s; return self
    def add(self, t):
        self.toppings.append(t); return self
    def build(self):
        return Pizza(self.size, self.toppings)

pizza = (PizzaBuilder()
         .set_size('L').add('cheese').add('mushroom')
         .build())
Mẹo PythonicTrong Python nhiều khi không cần Builder — dùng tham số mặc định / keyword args hoặc @dataclass là đủ. Builder sáng giá khi việc dựng có nhiều bước có thứ tự/điều kiện.

CH 8 Prototype Creational

Tạo đối tượng mới bằng cách sao chép (clone) một đối tượng mẫu có sẵn, thay vì new từ đầu.

🎯 Ý đồ:Khi khởi tạo tốn kém hoặc cấu hình phức tạp — copy một bản mẫu rồi chỉnh vài chỗ.
🔎 Dấu hiệu nhận biết Prototype
  • Có method clone() / copy() trả về bản sao cùng kiểu.
  • Tạo object mới bằng sao chép object có sẵn, không gọi constructor đầy đủ.
  • Hay gặp khi cần nhiều object gần giống nhau (game entity, cấu hình mặc định).
<?php
class Enemy {
    public function __construct(public int $hp, public array $skills) {}
    public function clone(): Enemy {                // PROTOTYPE: tự nhân bản
        return new Enemy($this->hp, $this->skills); // copy giá trị
    }
}
$orc = new Enemy(100, ['slash']);
$orc2 = $orc->clone();   // clone rồi chỉnh, khỏi dựng lại từ đầu
// PHP còn có từ khoá `clone $orc` + __clone() để deep copy
typedef struct { int hp; int skills[8]; int n; } Enemy;

Enemy enemy_clone(const Enemy* src) {   // PROTOTYPE: copy struct
    Enemy dst = *src;                   // sao chép từng byte (shallow)
    return dst;                         // con trỏ bên trong thì cần copy sâu thủ công
}
Enemy orc = { 100, {1}, 1 };
Enemy orc2 = enemy_clone(&orc);
import copy

class Enemy:
    def __init__(self, hp, skills):
        self.hp, self.skills = hp, skills

orc  = Enemy(100, ['slash'])
orc2 = copy.deepcopy(orc)    # PROTOTYPE: clone sâu, kể cả list bên trong
orc2.hp = 80                 # chỉnh bản sao, bản gốc không đổi
Bẫy shallow vs deep copyCopy "nông" (shallow) chỉ sao tham chiếu — sửa list trong bản sao làm hỏng cả bản gốc. Khi nhận diện Prototype, để ý nó dùng deep copy hay không (PHP __clone(), Python copy.deepcopy, C phải copy con trỏ thủ công).
ST · Cấu trúc (Structural)
Nhóm StructuralTất cả nói về cách ghép đối tượng thành cấu trúc lớn hơn. Dấu hiệu chung: có một lớp "bọc" (wrap) lớp khác — giữ tham chiếu tới đối tượng được bọc và chuyển tiếp/biến đổi lời gọi. Khác nhau ở mục đích của việc bọc.

CH 9 Adapter Structural

Bọc một lớp có giao diện "lệch" để nó khớp với interface mà code của bạn mong đợi.

🎯 Ý đồ:"Ổ cắm chuyển đổi". Cho hai thứ vốn không khớp nhau làm việc cùng nhau — thường để xài thư viện ngoài/legacy.
🔎 Dấu hiệu nhận biết Adapter
  • Một lớp XxxAdapter implements interface bạn cần, bên trong gọi xuống một đối tượng có API khác.
  • Method của adapter chủ yếu là dịch tên/định dạng: request() gọi legacy.specificRequest().
  • Mục đích: tái dùng code có sẵn mà không sửa nó, hoặc đổi cách gọi cho thống nhất.
<?php
interface Logger { public function log(string $msg): void; }

class ThirdPartyLib {                         // thư viện ngoài, API "lệch"
    public function writeLine(string $text) { /* ... */ }
}
class LibAdapter implements Logger {          // ADAPTER
    public function __construct(private ThirdPartyLib $lib) {}
    public function log(string $msg): void {
        $this->lib->writeLine("[LOG] $msg");  // dịch log() -> writeLine()
    }
}
function run(Logger $l){ $l->log("hi"); }     // code chỉ biết interface Logger
run(new LibAdapter(new ThirdPartyLib()));
typedef struct { void (*log)(void*, const char*); void* self; } Logger;   // interface mong đợi

void thirdparty_writeline(void* lib, const char* text){ /* ... */ }

// ADAPTER: hàm khớp chữ ký Logger, bên trong gọi API "lệch"
static void adapter_log(void* lib, const char* msg){
    char buf[256]; snprintf(buf,sizeof buf,"[LOG] %s",msg);
    thirdparty_writeline(lib, buf);          // dịch log -> writeline
}
Logger make_adapter(void* lib){ Logger l = { adapter_log, lib }; return l; }
class ThirdPartyLib:                  # API "lệch"
    def write_line(self, text): ...

class LibAdapter:                     # ADAPTER
    def __init__(self, lib):
        self._lib = lib
    def log(self, msg):               # interface mà code ta mong đợi
        self._lib.write_line(f"[LOG] {msg}")   # dịch log -> write_line

def run(logger):                      # duck typing: chỉ cần có .log()
    logger.log("hi")
run(LibAdapter(ThirdPartyLib()))
Adapter vs Decorator vs ProxyCả ba đều "bọc". Khác ở ý đồ: Adapter đổi giao diện (cho khớp); Decorator giữ nguyên giao diện nhưng thêm hành vi; Proxy giữ nguyên giao diện và kiểm soát truy cập (lazy, cache, quyền). Nhìn "vì sao bọc" để phân biệt.

CH 10 Decorator Structural

Bọc một đối tượng để thêm hành vi mà không sửa lớp gốc — và có thể bọc chồng nhiều lớp.

🎯 Ý đồ:Thêm tính năng linh hoạt lúc chạy bằng cách "lồng" các lớp cùng interface vào nhau (như mặc nhiều áo).
🔎 Dấu hiệu nhận biết Decorator
  • Lớp decorator cùng interface với đối tượng nó bọc, và giữ một tham chiếu tới đối tượng đó.
  • Method gọi $inner->op() rồi thêm thắt trước/sau (thêm dữ liệu, ghi log, nén...).
  • Có thể bọc chồng: new Gzip(new Encrypt(new File())).
<?php
interface Coffee { public function cost(): int; }
class Espresso implements Coffee { public function cost(): int { return 20; } }

abstract class CoffeeDecorator implements Coffee {     // DECORATOR base
    public function __construct(protected Coffee $inner) {}  // giữ đối tượng bị bọc
}
class WithMilk extends CoffeeDecorator {
    public function cost(): int { return $this->inner->cost() + 5; }  // thêm hành vi
}
class WithSugar extends CoffeeDecorator {
    public function cost(): int { return $this->inner->cost() + 2; }
}
$c = new WithSugar(new WithMilk(new Espresso()));  // bọc chồng -> 27
typedef struct Coffee { int (*cost)(struct Coffee*); struct Coffee* inner; } Coffee;

int espresso_cost(Coffee* c){ return 20; }
int milk_cost(Coffee* c){ return c->inner->cost(c->inner) + 5; }   // gọi inner rồi +5
int sugar_cost(Coffee* c){ return c->inner->cost(c->inner) + 2; }

Coffee espresso = { espresso_cost, NULL };
Coffee milk     = { milk_cost,  &espresso };   // DECORATOR: bọc espresso
Coffee sugar    = { sugar_cost, &milk };       // bọc chồng -> 27
class Espresso:
    def cost(self): return 20

class CoffeeDecorator:                 # DECORATOR
    def __init__(self, inner):
        self._inner = inner            # giữ đối tượng bị bọc

class WithMilk(CoffeeDecorator):
    def cost(self): return self._inner.cost() + 5
class WithSugar(CoffeeDecorator):
    def cost(self): return self._inner.cost() + 2

c = WithSugar(WithMilk(Espresso()))    # bọc chồng -> 27
Lưu ý "decorator" của PythonCú pháp @my_decorator trong Python cũng là hiện thân của pattern này (bọc một hàm để thêm hành vi) — nhưng ở cấp hàm, không phải đối tượng. Cùng ý tưởng "bọc thêm hành vi".

CH 11 Facade Structural

Cung cấp một giao diện đơn giản che giấu cả một hệ thống con phức tạp phía sau.

🎯 Ý đồ:"Một nút bấm" cho khách hàng. Gom nhiều bước/nhiều lớp con thành một method dễ dùng.
🔎 Dấu hiệu nhận biết Facade
  • Một lớp "cổng" với method đơn giản (placeOrder()) bên trong điều phối nhiều lớp con.
  • Client chỉ nói chuyện với facade, không đụng các lớp con bên dưới.
  • Facade không thêm hành vi mới, chỉ đơn giản hoá & sắp xếp lời gọi.
<?php
class OrderFacade {                        // FACADE
    public function __construct(
        private Inventory $inv,
        private Payment $pay,
        private Shipping $ship
    ) {}
    public function placeOrder(int $item, int $qty): void {  // 1 method giấu 3 hệ thống con
        $this->inv->reserve($item, $qty);
        $this->pay->charge($item * $qty);
        $this->ship->schedule($item, $qty);
    }
}
$order->placeOrder(42, 2);   // client không cần biết Inventory/Payment/Shipping
// FACADE: một hàm gom các hệ con
void place_order(Inventory* inv, Payment* pay, Shipping* ship,
                 int item, int qty) {
    inventory_reserve(inv, item, qty);
    payment_charge(pay, item * qty);
    shipping_schedule(ship, item, qty);
}
// caller chỉ gọi place_order(...), không cần biết 3 module bên trong
class OrderFacade:                    # FACADE
    def __init__(self, inv, pay, ship):
        self.inv, self.pay, self.ship = inv, pay, ship
    def place_order(self, item, qty):  # 1 method giấu nhiều hệ con
        self.inv.reserve(item, qty)
        self.pay.charge(item * qty)
        self.ship.schedule(item, qty)

order.place_order(42, 2)   # client không cần biết bên trong
Facade vs AdapterCả hai đơn giản hoá việc gọi. Khác: Adapter làm cho một lớp khớp một interface cụ thể; Facade gom nhiều lớp thành một API tiện lợi mới (không nhằm khớp interface có sẵn nào).

CH 12 Composite Structural

Cho phép xử lý đối tượng đơn lẻ và nhóm đối tượng giống hệt nhau — cấu trúc cây.

🎯 Ý đồ:"Lá" và "cành" có cùng interface, nên client duyệt cây mà không cần phân biệt. Vd: file/thư mục, menu/menu con.
🔎 Dấu hiệu nhận biết Composite
  • Một interface chung cho cả phần tử lá và phần tử chứa (group).
  • Phần tử chứa có danh sách con cùng kiểu, và method của nó đệ quy gọi xuống con.
  • Client gọi render()/size() lên gốc mà không cần biết là lá hay nhóm.
<?php
interface Node { public function size(): int; }

class File implements Node {                 // LÁ
    public function __construct(private int $bytes) {}
    public function size(): int { return $this->bytes; }
}
class Folder implements Node {               // COMPOSITE (cành)
    private array $children = [];
    public function add(Node $n): void { $this->children[] = $n; }
    public function size(): int {
        return array_sum(array_map(fn($c) => $c->size(), $this->children));  // đệ quy
    }
}
// folder->size() cộng dồn cả cây, không cần phân biệt file hay folder
typedef struct Node { int (*size)(struct Node*); } Node;

typedef struct { Node base; int bytes; } File;
int file_size(Node* n){ return ((File*)n)->bytes; }    // LÁ

typedef struct { Node base; Node* kids[16]; int n; } Folder;
int folder_size(Node* n){                               // COMPOSITE
    Folder* f = (Folder*)n; int total = 0;
    for (int i=0;i<f->n;i++) total += f->kids[i]->size(f->kids[i]);  // đệ quy
    return total;
}
class File:                          # LÁ
    def __init__(self, bytes_): self.bytes = bytes_
    def size(self): return self.bytes

class Folder:                        # COMPOSITE
    def __init__(self): self.children = []
    def add(self, node): self.children.append(node)
    def size(self):
        return sum(c.size() for c in self.children)   # đệ quy xuống con

# root.size() duyệt cả cây như nhau, không phân biệt lá/nhóm

CH 13 Proxy Structural

Một "người đại diện" cùng interface, đứng trước đối tượng thật để kiểm soát truy cập.

🎯 Ý đồ:Chèn logic trước/sau khi chạm vào đối tượng thật: tải lười (lazy), cache, kiểm tra quyền, ghi log, gọi từ xa.
🔎 Dấu hiệu nhận biết Proxy
  • Lớp XxxProxy cùng interface với đối tượng thật, giữ tham chiếu (hoặc tạo lười) tới nó.
  • Trước khi chuyển tiếp lời gọi, nó làm thêm việc: kiểm tra quyền / cache / khởi tạo lười / log.
  • Khác Decorator: Proxy quản lý vòng đời/truy cập chứ không "thêm tính năng nghiệp vụ".
<?php
interface Image { public function display(): void; }

class RealImage implements Image {           // đối tượng thật, tải nặng
    public function __construct(private string $file) { $this->load(); }
    private function load(): void { /* đọc đĩa, tốn kém */ }
    public function display(): void { /* ... */ }
}
class ImageProxy implements Image {          // PROXY tải lười
    private ?RealImage $real = null;
    public function __construct(private string $file) {}
    public function display(): void {
        $this->real ??= new RealImage($this->file);  // chỉ tải khi thật sự cần
        $this->real->display();
    }
}
typedef struct { void (*display)(void*); void* self; } Image;

typedef struct { const char* file; RealImage* real; } ImageProxy;   // PROXY
void proxy_display(void* p_){
    ImageProxy* p = p_;
    if (!p->real)                          // tải lười: tạo lần đầu
        p->real = real_image_load(p->file);
    real_image_display(p->real);           // rồi mới uỷ quyền cho đối tượng thật
}
class RealImage:                       # đối tượng thật, tải nặng
    def __init__(self, file):
        self.file = file; self._load()
    def _load(self): ...               # đọc đĩa, tốn kém
    def display(self): ...

class ImageProxy:                      # PROXY tải lười
    def __init__(self, file):
        self.file, self._real = file, None
    def display(self):
        if self._real is None:         # chỉ tải khi cần
            self._real = RealImage(self.file)
        self._real.display()
Proxy ngoài đờiORM "lazy loading" (Doctrine, SQLAlchemy), @property cache, Cloudflare đứng trước web server, RPC stub gọi service từ xa — đều là Proxy. Thấy "một lớp trung gian kiểm soát truy cập tới thứ thật" là nó.

CH 13+ Bridge & Flyweight Structural

Hai pattern cấu trúc ít gặp hơn — nắm dấu hiệu để nhận ra khi gặp.

🌉 Bridge ST

Tách abstraction khỏi implementation để hai phía biến đổi độc lập.
Dấu hiệu: một lớp "trừu tượng" (Remote) giữ tham chiếu tới một interface "hiện thực" (Device) và uỷ quyền sang đó. Tránh bùng nổ tổ hợp lớp con (TV+Sony, TV+LG, Radio+Sony...) bằng cách ghép 2 trục qua composition thay vì kế thừa.

🪶 Flyweight ST

Chia sẻ phần dữ liệu chung (intrinsic) giữa nhiều đối tượng để tiết kiệm bộ nhớ.
Dấu hiệu: có một factory/cache trả về object dùng chung thay vì tạo mới; trạng thái được tách thành phần "chung được chia sẻ" và phần "riêng truyền vào lúc gọi". Vd: glyph của font, tile của map game, đối tượng màu sắc.
Nhận diện nhanhThấy "một getX() trả về object đã tồn tại trong pool/cache thay vì new" → Flyweight. Thấy "hai phân cấp song song nối nhau bằng một tham chiếu interface" → Bridge.
BE · Hành vi (Behavioral)
Nhóm BehavioralTất cả nói về cách các đối tượng giao tiếp & phân chia trách nhiệm: ai gọi ai, thuật toán nào chạy, sự kiện lan truyền ra sao. Đây là nhóm dễ nhầm nhất vì nhiều pattern có cấu trúc giống nhau — phân biệt bằng ý đồ.

CH 14 Strategy Behavioral

Đóng gói nhiều thuật toán thành các lớp hoán đổi được, chọn lúc chạy.

🎯 Ý đồ:"Cùng một việc, nhiều cách làm". Thay if/else chọn thuật toán bằng các đối tượng strategy cắm vào.
🔎 Dấu hiệu nhận biết Strategy
  • Một interface với một method (vd execute(), sort(), pay()) và nhiều lớp hiện thực khác nhau.
  • Một lớp "context" giữ tham chiếu strategy và cho phép đổi nó lúc chạy (setStrategy()).
  • Thường thay thế một chuỗi if/switch chọn cách xử lý.
<?php
interface PayStrategy { public function pay(int $amount): void; }   // 1 method, nhiều cách
class PayPal  implements PayStrategy { public function pay(int $a): void { /*...*/ } }
class Crypto  implements PayStrategy { public function pay(int $a): void { /*...*/ } }

class Checkout {                              // CONTEXT
    public function __construct(private PayStrategy $strategy) {}
    public function setStrategy(PayStrategy $s): void { $this->strategy = $s; }  // đổi lúc chạy
    public function process(int $amount): void { $this->strategy->pay($amount); }
}
$c = new Checkout(new PayPal());
$c->setStrategy(new Crypto());   // hoán đổi thuật toán mà không sửa Checkout
// STRATEGY trong C = một con trỏ hàm
typedef void (*PayStrategy)(int amount);

void pay_paypal(int a){ /* ... */ }
void pay_crypto(int a){ /* ... */ }

typedef struct { PayStrategy pay; } Checkout;     // CONTEXT giữ strategy
void checkout_process(Checkout* c, int amount){ c->pay(amount); }

Checkout c = { pay_paypal };
c.pay = pay_crypto;          // đổi thuật toán lúc chạy
checkout_process(&c, 100);
# Python: strategy thường chỉ là một hàm (first-class function)
def pay_paypal(amount): ...
def pay_crypto(amount): ...

class Checkout:                       # CONTEXT
    def __init__(self, strategy):
        self.strategy = strategy
    def process(self, amount):
        self.strategy(amount)         # gọi strategy đang cắm

c = Checkout(pay_paypal)
c.strategy = pay_crypto               # hoán đổi lúc chạy
c.process(100)
Strategy vs State — cặp song sinhCấu trúc gần như giống hệt. Khác ở ý đồ: Strategy = client chủ động chọn một cách làm và nó thường không tự đổi; State = đối tượng tự chuyển trạng thái khiến hành vi đổi theo. Xem CH17.

CH 15 Observer Behavioral

Khi một đối tượng đổi trạng thái, tự động thông báo cho mọi đối tượng đăng ký theo dõi.

🎯 Ý đồ:Quan hệ "một-tới-nhiều": publisher không cần biết ai nghe, chỉ notify(). Nền tảng của event/pub-sub.
🔎 Dấu hiệu nhận biết Observer
  • Có subscribe()/attach()/on() và notify()/emit(); một danh sách observer.
  • notify() lặp qua danh sách gọi update() trên từng observer.
  • Publisher (subject) không biết lớp cụ thể của observer — chỉ biết interface.
<?php
interface Observer { public function update(string $event): void; }

class Subject {                               // PUBLISHER
    private array $observers = [];
    public function subscribe(Observer $o): void { $this->observers[] = $o; }   // đăng ký
    public function notify(string $event): void {
        foreach ($this->observers as $o) $o->update($event);   // lặp & báo cho tất cả
    }
}
class EmailAlert implements Observer {
    public function update(string $e): void { /* gửi mail */ }
}
$s = new Subject();
$s->subscribe(new EmailAlert());
$s->notify('order.created');   // mọi observer được gọi
typedef void (*Observer)(const char* event);

typedef struct { Observer list[16]; int n; } Subject;   // PUBLISHER
void subscribe(Subject* s, Observer o){ s->list[s->n++] = o; }
void notify(Subject* s, const char* event){
    for (int i=0;i<s->n;i++) s->list[i](event);   // báo cho tất cả
}
void email_alert(const char* e){ /* gửi mail */ }

Subject s = {0};
subscribe(&s, email_alert);
notify(&s, "order.created");
class Subject:                        # PUBLISHER
    def __init__(self):
        self._observers = []
    def subscribe(self, fn):          # observer có thể chỉ là callback
        self._observers.append(fn)
    def notify(self, event):
        for obs in self._observers:   # lặp & báo cho tất cả
            obs(event)

s = Subject()
s.subscribe(lambda e: print("mail:", e))
s.notify("order.created")
Observer ở khắp nơiEvent listener của JS/DOM, signal của Qt/Godot, Laravel Events, Django signals, hệ thống pub/sub (Redis, Kafka ở mức ý tưởng) — đều là Observer. Thấy "đăng ký rồi được gọi lại khi có chuyện" là nó.

CH 16 Command Behavioral

Đóng gói một yêu cầu/hành động thành đối tượng — để xếp hàng, ghi log, undo/redo.

🎯 Ý đồ:Biến "lời gọi method" thành object có thể truyền đi, lưu lại, hoàn tác. Tách người ra lệnh khỏi người thực hiện.
🔎 Dấu hiệu nhận biết Command
  • Một interface với method execute() (và thường undo()); mỗi hành động là một lớp command.
  • Có nơi lưu/xếp hàng/lặp lại các command (history stack, queue).
  • Đối tượng command giữ sẵn tham số & receiver để chạy sau, không chạy ngay.
<?php
interface Command { public function execute(): void; public function undo(): void; }

class AddText implements Command {            // 1 hành động = 1 object
    public function __construct(private Editor $ed, private string $text) {}
    public function execute(): void { $this->ed->append($this->text); }
    public function undo(): void    { $this->ed->removeLast($this->text); }
}
class History {                               // xếp hàng để undo
    private array $stack = [];
    public function run(Command $c): void { $c->execute(); $this->stack[] = $c; }
    public function undo(): void { (array_pop($this->stack))?->undo(); }
}
typedef struct Command {
    void (*execute)(struct Command*);
    void (*undo)(struct Command*);
    void* receiver; const char* arg;          // giữ sẵn dữ liệu để chạy sau
} Command;

typedef struct { Command* stack[64]; int n; } History;
void history_run(History* h, Command* c){ c->execute(c); h->stack[h->n++] = c; }
void history_undo(History* h){ if(h->n){ Command* c=h->stack[--h->n]; c->undo(c); } }
class AddText:                        # 1 hành động = 1 object
    def __init__(self, editor, text):
        self.editor, self.text = editor, text
    def execute(self): self.editor.append(self.text)
    def undo(self):    self.editor.remove_last(self.text)

class History:                        # stack để undo
    def __init__(self): self.stack = []
    def run(self, cmd):
        cmd.execute(); self.stack.append(cmd)
    def undo(self):
        if self.stack: self.stack.pop().undo()
Nhận ra Command khi...thấy nút Undo/Redo, hàng đợi job (queue worker), macro/replay, hay "menu item nào cũng có một object hành động gắn vào". Khác Strategy: Strategy = cách làm một việc; Command = một yêu cầu đã đóng gói (kèm dữ liệu, để chạy/hoãn/hoàn tác).

CH 17 State Behavioral

Đối tượng thay đổi hành vi khi trạng thái nội bộ đổi — như thể đổi sang lớp khác.

🎯 Ý đồ:Thay khối if (state == ...) khổng lồ bằng các lớp trạng thái; mỗi trạng thái tự xử lý & tự quyết chuyển sang trạng thái nào.
🔎 Dấu hiệu nhận biết State
  • Một interface State với các method hành vi; nhiều lớp trạng thái cụ thể (DraftState, PublishedState).
  • Context giữ trạng thái hiện tại và uỷ quyền cho nó; chính state tự gọi context.setState(...) để chuyển.
  • Bản chất là một máy trạng thái (FSM) được object-hoá.
<?php
interface State { public function next(Doc $d): void; public function name(): string; }

class Draft implements State {
    public function next(Doc $d): void { $d->setState(new Published()); }  // state TỰ chuyển tiếp
    public function name(): string { return 'draft'; }
}
class Published implements State {
    public function next(Doc $d): void { /* đã cuối, không đổi */ }
    public function name(): string { return 'published'; }
}
class Doc {                                   // CONTEXT
    private State $state;
    public function __construct(){ $this->state = new Draft(); }
    public function setState(State $s): void { $this->state = $s; }
    public function publish(): void { $this->state->next($this); }  // hành vi đổi theo state
}
struct Doc;
typedef struct { void (*next)(struct Doc*); const char* name; } State;

void draft_next(struct Doc* d);
void published_next(struct Doc* d){ /* cuối, không đổi */ }
State DRAFT     = { draft_next, "draft" };
State PUBLISHED = { published_next, "published" };

typedef struct Doc { State* state; } Doc;     // CONTEXT
void draft_next(Doc* d){ d->state = &PUBLISHED; }   // state tự chuyển
void doc_publish(Doc* d){ d->state->next(d); }
class Draft:
    def next(self, doc): doc.state = Published()   # state tự chuyển tiếp
    name = 'draft'
class Published:
    def next(self, doc): pass                      # trạng thái cuối
    name = 'published'

class Doc:                            # CONTEXT
    def __init__(self): self.state = Draft()
    def publish(self): self.state.next(self)   # hành vi đổi theo state
State vs Strategy — cách phân biệt dứt khoátNhìn ai đổi và vì sao: nếu các "cách làm" không biết về nhau và do client chọn → Strategy. Nếu các lớp tự chuyển sang nhau theo luồng (A→B→C) và mô tả "trạng thái" → State.

CH 18 Template Method Behavioral

Lớp cha định khung thuật toán, để lớp con điền vào vài bước cụ thể.

🎯 Ý đồ:"Bộ xương cố định, vài chỗ trống cho lớp con". Tái dùng cấu trúc chung, tuỳ biến chi tiết.
🔎 Dấu hiệu nhận biết Template Method
  • Một method không-abstract ở lớp cha gọi tuần tự nhiều bước, trong đó vài bước là abstract.
  • Lớp con chỉ override các bước, không đổi trình tự tổng thể.
  • Dựa trên kế thừa (khác Strategy dùng composition).
<?php
abstract class DataExporter {
    public final function export(): void {     // TEMPLATE METHOD: khung cố định
        $data = $this->fetch();                 // bước chung
        $out  = $this->format($data);           // bước do lớp con quyết
        $this->save($out);                      // bước chung
    }
    protected function fetch(): array { return [1,2,3]; }
    abstract protected function format(array $d): string;   // "chỗ trống"
    protected function save(string $o): void { /* ghi file */ }
}
class CsvExporter extends DataExporter {
    protected function format(array $d): string { return implode(',', $d); }  // chỉ điền 1 bước
}
typedef struct Exporter {
    const char* (*format)(struct Exporter*, int* data, int n);   // "chỗ trống" (hook)
} Exporter;

void export_run(Exporter* e){                  // TEMPLATE METHOD: khung cố định
    int data[] = {1,2,3};
    const char* out = e->format(e, data, 3);    // gọi bước tuỳ biến
    save_to_file(out);                          // bước chung
}
const char* csv_format(Exporter* e, int* d, int n){ /* "1,2,3" */ return "1,2,3"; }
Exporter csv = { csv_format };
from abc import ABC, abstractmethod

class DataExporter(ABC):
    def export(self):                 # TEMPLATE METHOD: khung cố định
        data = self.fetch()
        out  = self.format(data)      # bước do lớp con quyết
        self.save(out)
    def fetch(self): return [1, 2, 3]
    @abstractmethod
    def format(self, data): ...       # "chỗ trống"
    def save(self, out): ...

class CsvExporter(DataExporter):
    def format(self, data):           # chỉ override 1 bước
        return ",".join(map(str, data))
Template Method vs StrategyCùng mục tiêu "tuỳ biến một phần thuật toán". Template Method dùng kế thừa (cố định lúc biên dịch, override bước); Strategy dùng composition (đổi cả thuật toán lúc chạy). Thấy abstract + lớp con override vài bước → Template Method.

CH 19 Iterator Behavioral

Duyệt lần lượt các phần tử của một tập hợp mà không lộ cấu trúc bên trong.

🎯 Ý đồ:Tách "cách duyệt" khỏi "cách lưu trữ". Cho phép foreach/for..in trên cấu trúc tuỳ ý.
🔎 Dấu hiệu nhận biết Iterator
  • Có cặp hasNext()/next() hoặc method tạo iterator (__iter__, IteratorAggregate).
  • Cho phép duyệt nhiều lần / nhiều kiểu duyệt trên cùng tập hợp mà không phơi mảng nội bộ.
  • Trong PHP/Python/Java thường là tích hợp ngôn ngữ (interface chuẩn) hơn là tự viết.
<?php
class NumberCollection implements IteratorAggregate {   // interface chuẩn của PHP
    private array $items = [];
    public function add(int $n): void { $this->items[] = $n; }
    public function getIterator(): Iterator {
        return new ArrayIterator($this->items);   // trả iterator, giấu mảng
    }
}
$c = new NumberCollection();
$c->add(1); $c->add(2);
foreach ($c as $n) { /* duyệt mà không chạm mảng nội bộ */ }
// C: iterator = struct giữ vị trí + hàm next
typedef struct { int* data; int n; int pos; } Iter;
int  iter_has_next(Iter* it){ return it->pos < it->n; }
int  iter_next(Iter* it){ return it->data[it->pos++]; }

int nums[] = {1,2,3};
Iter it = { nums, 3, 0 };
while (iter_has_next(&it)) {
    int v = iter_next(&it);   // duyệt qua interface, không cần biết "data" là mảng
}
class NumberCollection:
    def __init__(self): self._items = []
    def add(self, n): self._items.append(n)
    def __iter__(self):               # giao thức iterator của Python
        return iter(self._items)      # giấu cấu trúc nội bộ

c = NumberCollection(); c.add(1); c.add(2)
for n in c:                           # for..in dùng __iter__
    ...
# generator cũng là Iterator: def gen(): yield 1; yield 2
Bạn đã dùng nó hằng ngàyforeach (PHP), for..in + generator (Python), Iterator (Java/STL) — đều là pattern Iterator được ngôn ngữ chuẩn hoá. Hiếm khi cần tự viết, nhưng nhận ra nó giúp hiểu vì sao mọi tập hợp đều duyệt được như nhau.

CH 20 Chain of Responsibility Behavioral

Chuyền một yêu cầu qua chuỗi các handler; mỗi handler xử lý hoặc đẩy tiếp.

🎯 Ý đồ:Tách người gửi khỏi người xử lý; nhiều handler có cơ hội xử lý theo thứ tự (vd middleware, validation, escalation).
🔎 Dấu hiệu nhận biết Chain of Responsibility
  • Mỗi handler giữ tham chiếu tới next handler, và quyết định tự xử lý hay chuyển tiếp.
  • Yêu cầu "đi xuyên qua" một danh sách/chuỗi cho tới khi ai đó xử lý (hoặc hết chuỗi).
  • Hay gặp ở middleware (HTTP), pipeline xử lý, bộ lọc, hệ thống duyệt nhiều cấp.
<?php
abstract class Handler {
    private ?Handler $next = null;
    public function setNext(Handler $h): Handler { $this->next = $h; return $h; }
    public function handle(int $level): void {
        if (!$this->process($level) && $this->next)   // không xử lý được -> đẩy tiếp
            $this->next->handle($level);
    }
    abstract protected function process(int $level): bool;
}
class Support extends Handler {
    protected function process(int $l): bool { return $l <= 1; }   // chỉ lo cấp thấp
}
class Manager extends Handler {
    protected function process(int $l): bool { return $l <= 3; }
}
$support = new Support();
$support->setNext(new Manager());     // dựng chuỗi
$support->handle(3);                   // Support bỏ qua -> Manager xử lý
typedef struct Handler {
    int (*process)(int level);
    struct Handler* next;             // con trỏ tới handler kế tiếp
} Handler;

void handle(Handler* h, int level){
    if (h->process(level)) return;    // xử lý được thì dừng
    if (h->next) handle(h->next, level);   // không thì đẩy tiếp
}
int support_proc(int l){ return l <= 1; }
int manager_proc(int l){ return l <= 3; }
Handler manager = { manager_proc, NULL };
Handler support = { support_proc, &manager };
handle(&support, 3);
class Handler:
    def __init__(self): self._next = None
    def set_next(self, h):
        self._next = h; return h
    def handle(self, level):
        if not self.process(level) and self._next:   # không xử lý -> đẩy tiếp
            self._next.handle(level)

class Support(Handler):
    def process(self, l): return l <= 1
class Manager(Handler):
    def process(self, l): return l <= 3

support = Support(); support.set_next(Manager())
support.handle(3)                     # Support bỏ qua -> Manager xử lý
Nhận ra qua "middleware"Express.js, Laravel middleware, Django middleware, ASP.NET pipeline — tất cả là Chain of Responsibility: request đi qua từng lớp, mỗi lớp xử lý phần của mình rồi gọi next().

CH 20+ Mediator · Memento · Visitor · Interpreter Behavioral

Bốn pattern hành vi còn lại — học dấu hiệu để nhận ra khi gặp.

🛫 Mediator BE

Một đối tượng trung tâm điều phối giao tiếp giữa nhiều đối tượng, để chúng không tham chiếu thẳng tới nhau.
Dấu hiệu: các component gọi mediator.notify(this, event) thay vì gọi nhau; logic "ai phản ứng với ai" nằm tập trung ở mediator. Vd: chat room, air traffic control, dialog form điều phối các widget.

💾 Memento BE

Lưu & khôi phục trạng thái nội bộ của một đối tượng mà không phá vỡ đóng gói.
Dấu hiệu: có save() trả về một "snapshot" mờ đục và restore(snapshot). Lõi của undo/redo, save game, checkpoint. (Thường đi cùng Command.)

🧳 Visitor BE

Thêm thao tác mới lên một cây đối tượng mà không sửa các lớp trong cây.
Dấu hiệu: các phần tử có accept(visitor) rồi gọi ngược visitor.visitX(this) (double dispatch). Vd: trình duyệt AST của compiler, xuất nhiều định dạng từ một cấu trúc.

📜 Interpreter BE

Biểu diễn ngữ pháp của một ngôn ngữ nhỏ thành cây lớp, mỗi lớp tự interpret().
Dấu hiệu: mỗi luật ngữ pháp = một lớp với interpret(context); cây biểu thức tự tính kết quả. Vd: máy tính biểu thức, bộ lọc rule, regex engine (ý tưởng).
